**Введение**

Современный образовательный ландшафт требует гибких и доступных решений, особенно в нишевом профессиональном обучении, таком как диджеинг. Тема данного курсового проекта — «Веб-приложение для начинающих диджеев». Актуальность работы обусловлена ростом популярности электронной музыки и увеличением спроса на качественные, структурированные онлайн-курсы, которые позволяют быстро освоить профессию, минуя дорогостоящие офлайн-школы. Создание специализированной платформы, объединяющей обучающий контент, тестирование и механизм обратной связи (отзывы и оценки), имеет высокую значимость для эффективного развития начинающих специалистов.  
Цель курсового проекта заключается в разработке функционально полного и логически завершенного веб\-приложения, способного обеспечить дистанционное обучение диджеев.  
Для достижения поставленной цели необходимо выполнить следующие задачи:

* спроектировать архитектуру веб\-приложения с асинхронным пользовательским интерфейсом (UI), обеспечивая максимальную независимость отображения, бизнес-логики и хранилища данных для возможности дальнейшего расширения;  
* разработать систему управления доступом с реализацией трех ролей: модератор, пользователь, гость, включая механизмы регистрации и авторизации;  
* реализовать функционал для пользователей, включающий просмотр содержимого купленных курсов, прохождение тестов, управление личным кабинетом и добавление отзывов/оценок;  
* создать интерфейс для модератора, позволяющий управлять курсами, блокировать пользователей, а также скрывать и удалять отзывы;  
* разработать механизмы поиска, фильтрации и сортировки контента для удобного использования.

Конечным продуктом работы станет веб\-приложение, которое выполнено с использованием асинхронного программирования, взаимодействует с базой данных и упаковано в Docker-контейнеры. Приложение будет иметь интуитивно понятное и удобное управление.  
В качестве технологической основы проекта будут использованы:

* язык разработки: JavaScript;  
* платформа: Node.Js;  
* проектирование: UML-диаграммы (вариантов использования и тд).

Ожидается, что разработанное веб\-приложение станет эффективной и масштабируемой образовательной платформой для освоения основ диджеинга.

2. **Разработка функциональных требований**

Разработка функциональных требований является основополагающим этапом проектирования, определяющим, что именно программное средство должно делать. Функциональные требования описывают поведение системы, ее сервисы, реакцию на определенные входные данные и ее поведение в конкретных ситуациях.  
Функционально веб\-приложение должно: 

1) обеспечивать возможность регистрации и авторизации;  
2) обеспечивать реализацию трёх ролей: модератор, пользователь, гость;  
3) предоставлять возможность модератору:  
   1) управления единицей контента (курс);  
   2) управления пользователями: блокировка;  
   3) предоставлять возможность модератору скрывать и удалять отзывы.   
4) предоставлять возможность поиска, фильтрации и сортировки единиц контента (курс);   
5) предоставлять возможность пользователю:  
   1) добавлять отзывы и оценки к единице контента;  
   2) управлять личным кабинетом: редактирование, удаление, просмотр истории покупок и список текущих курсов;  
   3) совершить покупку курса и получить подтверждающее письмо на почту;  
   4) просматривать содержимое купленного курса и проходить тесты на основе пройденной информации.   
6) предоставить возможность гостю просматривать общедоступную информацию на сайте, список курсов и их описание.

Листинг 3.1 – Фрагмент функции регистрации пользователя

Система получает из тела запроса имя, email и пароль. Перед созданием записи выполняется проверка на существование пользователя с таким же email. Для обеспечения безопасности пароль не хранится в открытом виде, а хэшируется с использованием алгоритма Bcrypt. После успешного создания пользователя в базе данных генерируется JWT-токен, который возвращается клиенту для последующей аутентификации.  
Процесс авторизации пользователя в системе реализован на серверной части и инициируется, когда клиентское приложение отправляет POST-запрос на эндпоинт /api/auth/login. Серверный контроллер authController получает из тела запроса учетные данные пользователя — email и password. Первым шагом является поиск пользователя в базе данных MongoDB по его уникальному email. Если пользователь не найден, система немедленно возвращает ошибку "Неверные учетные данные".  
Процесс авторизации в контроллере authController представлен в листинге 3.2.

| const loginUser \= async (req, res) \=\> {   try {     const { email, password } \= req.body;     const user \= await User.findOne({ email });     if (user && (await bcrypt.compare(password, user.password))) {       if (user.isBlocked) {         return res.status(403).json({ message: 'Пользователь заблокирован' });       }       res.json({         \_id: user.id,         username: user.username,         email: user.email,         role: user.role,         isBlocked: user.isBlocked || false,         token: generateToken(user.\_id),       });     } else {       res.status(400).json({ message: 'Неверные учетные данные' });     }  }; |

Листинг 3.2 – Процесс авторизации пользователя

В случае, если пользователь найден, выполняется ключевой этап проверки безопасности: сравнение паролей. Для этого используется асинхронная функция bcrypt.compare(), которая сравнивает хэш пароля. После успешной проверки пароля система дополнительно проверяет, не заблокирован ли аккаунт пользователя (isBlocked). Если все проверки пройдены, с помощью библиотеки jsonwebtoken генерируется JWT-токен.   
На клиентской стороне, в сервисе authService, асинхронная функция login отправляет запрос и ожидает ответа от сервера. После получения успешного ответа, содержащего данные пользователя и сгенерированный токен, весь полученный объект пользователя сериализуется в строку формата JSON и сохраняется в localStorage браузера.   
Для разграничения доступа к приватным ресурсам API, таким как данные профиля пользователя, реализована функция промежуточной обработки (middleware) protect. Она выступает в роли «стража» и выполняется перед основной логикой каждого защищенного маршрута, перехватывая все входящие запросы.  
Алгоритм начинается с проверки наличия заголовка Authorization в запросе и его соответствия стандартному формату. Если заголовок присутствует, из него извлекается сам JWT-токен. Затем, с помощью метода jwt.verify() и секретного ключа сервера, хранящегося в переменных окружения, выполняется валидация токена.  
Реализация разграничения доступа представлена в листинге 3.3 на примере перехода на страницу профиля.

| const protect \= async (req, res, next) \=\> { let token; if (req.headers.authorization && req.headers.authorization.startsWith('Bearer')) {     try {         token \= req.headers.authorization.split(' ')\[1\];         const decoded \= jwt.verify(token, process.env.JWT\_SECRET);         req.user \= await User.findById(decoded.id).select('-password');         if (req.user && req.user.isBlocked) {             return res.status(403).json({ message: 'Аккаунт заблокирован' });         }         next();     } catch (error) {         res.status(401).json({ message: 'Нет авторизации, токен недействителен' });     } } if (\!token) {     res.status(401).json({ message: 'Нет авторизации, нет токена' }); } }; |
| :---- |

Листинг 3.3 – Реализация разграничения доступа

При успешной валидации токена, из его полезной нагрузки (payload) извлекается id пользователя. Этот идентификатор используется для запроса к базе данных с целью получения актуальных данных о пользователе. Дополнительно реализована проверка безопасности: система проверяет флаг isBlocked у пользователя. Если аккаунт заблокирован, доступ запрещается с отправкой статуса 403 Forbidden. Если же все проверки пройдены, объект пользователя прикрепляется к объекту запроса (req.user), и с помощью вызова функции next() управление передается следующему обработчику в цепочке – основной логике контроллера. Таким образом, обеспечивается надежная, многоуровневая защита API.

Для выполнения самого асинхронного запроса применяется хук useEffect. Он позволяет выполнять "побочные эффекты" в функциональных компонентах, к которым и относятся сетевые запросы. **3.3 Контейнеризация приложения с использованием Docker**

Для обеспечения стабильности, переносимости и упрощения процесса развертывания в проекте была применена технология контейнеризации Docker. Данный подход гарантирет, что приложение будет вести себя одинаково в любой среде — будь то компьютер разработчика, тестовый стенд или производственный сервер. Для этого каждый ключевой компонент системы — серверная часть (Backend на Node.js), клиентская часть (Frontend на React) и база данных (MongoDB) — был упакован в собственный изолированный контейнер. Такой контейнер включает в себя не только код приложения, но и все необходимые для его работы зависимости, библиотеки и системные настройки.  
Сервер приложений (Backend) был помещен в контейнер, сборка которого описывается в файле Dockerfile. Этот файл содержит набор инструкций, по которым создается образ, включающий в себя необходимую версию Node.js, установку всех проектных зависимостей из файла package.json и копирование исходного кода самого приложения. Аналогичным образом был контейнеризирован и Frontend: в его Dockerfile описан двухэтапный процесс, где сначала происходит сборка статических файлов React-приложения (npm run build), а затем эти файлы помещаются в легковесный контейнер с веб\-сервером Nginx, который настроен для их эффективной раздачи.  
Для управления базой данных MongoDB также используется официальный Docker-образ. Это избавляет от необходимости ручной установки и сложной настройки СУБД на целевой машине. Ключевым аспектом здесь является использование именованных томов (Docker Volumes). Для контейнера с MongoDB был настроен том, который связывает директорию внутри контейнера, где хранятся данные, с директорией на файловой системе хост-машины. Это обеспечивает персистентность данных: вся информация о пользователях, курсах и покупках надежно сохраняется даже при остановке, перезапуске или обновлении контейнера с базой данных, предотвращая ее потерю.  
Для управления всеми тремя контейнерами как единым целым и для определения их взаимодействия используется инструмент оркестровки Docker Compose. Вся конфигурация системы описана в декларативном файле docker-compose.yml. В этом файле определены все три сервиса (backend, frontend, mongo), их настройки сборки, проброс портов для доступа извне, а также переменные окружения, необходимые для работы (например, строка подключения к БД). Благодаря Docker Compose, весь проект — бэкенд, фронтенд и база данных — запускается и останавливается одной простой командой (docker-compose up / docker-compose down), что делает процесс развертывания и разработки максимально простым и быстрым.

В рамках данного раздела была выполнена основная практическая часть курсового проекта — программная реализация веб\-приложения "DJify". Разделение на независимые компоненты — серверную часть (бэкенд) на Node.js/Express, клиентскую часть (фронтенд) на React и базу данных MongoDB — позволило обеспечить модульность и гибкость, что является ключевым для дальнейшей поддержки и масштабирования проекта.

Особое внимание было уделено безопасности: с помощью функций промежуточной обработки (middleware) была реализована надежная защита маршрутов, обеспечивающая разграничение доступа к приватной информации и контенту в зависимости от статуса аутентификации и прав пользователя.  
На клиентской стороне было разработано интерактивное одностраничное приложение (SPA), отвечающее за пользовательский интерфейс. Благодаря компонентному подходу библиотеки React, были созданы переиспользуемые UI-элементы (components) и полноценные страницы (pages), что позволило добиться единообразия и ускорить разработку. Вся логика взаимодействия с сервером была инкапсулирована в слое сервисов, а для навигации по приложению без перезагрузки страниц была задействована библиотека react-router-dom. Управление состоянием и выполнение асинхронных запросов реализовано с помощью хуков useState и useEffect, что соответствует современным стандартам разработки на React.  
Завершающим и важным этапом разработки стала контейнеризация всего приложения с использованием технологии Docker и инструмента оркестровки Docker Compose. Упаковка бэкенда, фронтенда и базы данных в изолированные контейнеры позволила создать единую, переносимую и предсказуемую среду выполнения. Создать файл .env в корне проекта с необходимыми переменными окружения для работы приложения 

**5.3.1 Подготовка к развёртыванию**

Перед началом развёртывания необходимо выполнить следующие шаги:

* установить Docker и Docker Compose на целевом устройстве;  
* убедиться, что файл docker-compose.yml и все необходимые директории проекта (включая сервер и базу данных) находятся в одном рабочем каталоге;  
* проверить наличие файла .env с необходимыми переменными окружения для работы приложения (например, данные для подключения к базе данных, порты и секреты).

**5.3.2 Структура развёртывания**

Конфигурация развертывания включает четыре основных сервиса. Клиентская часть представляет собой React приложение, которое собирается в статические файлы и раздается через Nginx. Серверная часть запускается в отдельном контейнере с использованием Dockerfile и взаимодействует с базой данных через внутреннюю сеть Docker. База данных MongoDB запускается в отдельном контейнере. 

**5.3.3 Запуск приложения**

Для развертывания приложения необходимо открыть терминал в корневом каталоге проекта и выполнить команду docker-compose up \--build. Эта команда соберет образы для всех сервисов и запустит контейнеры. После успешного запуска необходимо убедиться, что все сервисы работают корректно. Клиентская часть будет доступна на порту 80, серверная часть на порту 3000, база данных на порту 2717\. Приложение готово к использованию после успешного запуска всех контейнеров.

**Заключение**

В ходе выполнения курсового проекта была успешно решена задача по разработке полнофункционального веб\-приложения «DJify», предназначенного для обучения начинающих диджеев. Основной целью являлось создание интерактивной образовательной платформы, которая бы предоставляла пользователям доступ к учебным материалам с гибкой системой разграничения прав доступа.  
Для реализации проекта был выбран современный и востребованный стек технологий MERN (MongoDB, Express.js, React, Node.js). Серверная часть (бэкенд) была реализована на платформе Node.js с использованием фреймворка Express.js, что позволило создать гибкий и масштабируемый RESTful API. В качестве системы управления базами данных была выбрана нереляционная СУБД MongoDB, гибкая схема которой идеально подошла для хранения разнородной структуры учебных курсов. Клиентская часть (фронтенд) была разработана с использованием библиотеки React, что позволило создать быстрый и интерактивный одностраничный интерфейс (SPA).  
В рамках проекта были реализованы ключевые функциональные модули, необходимые для работы современного веб\-приложения. Разработана надежная система аутентификации и авторизации пользователей, включающая регистрацию, вход в систему и безопасное хранение паролей с использованием хэширования. Для управления сессиями был применен механизм на основе JWT-токенов. Особое внимание было уделено реализации сложной логики восстановления пароля через подтверждение по электронной почте, что повышает безопасность и удобство использования платформы.  
Главной функциональной частью приложения является система курсов с разграничением доступа. Реализована возможность для незарегистрированных пользователей (гостей) просматривать общий список и описание курсов, в то время как полный доступ к учебным материалам открывается только после авторизации и совершения «покупки». Был успешно разработан личный кабинет пользователя, где он может просматривать список приобретенных курсов, а также изменять свои персональные данные, такие как имя пользователя и аватар.  
В процессе работы над проектом были получены и закреплены ценные практические навыки в области веб\-разработки. Были преодолены такие сложности, как управление асинхронными операциями, организация взаимодействия между клиентом и сервером, управление состоянием в React-приложении и безопасная обработка пользовательских данных.  
Разработанное приложение является полнофункциональным прототипом. Таким образом, все цели и задачи, поставленные в задании, были выполнены, а созданный продукт готов к дальнейшему совершенствованию.
