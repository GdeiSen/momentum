# 4 Тестирование веб-приложения

## 4.1 Методология тестирования

Тестирование веб-приложения Momentum выполнено с использованием комплексного подхода, включающего несколько уровней проверки функциональности и качества системы. Основная цель тестирования заключается в обеспечении корректной работы всех компонентов приложения, выявлении и устранении ошибок, а также проверке соответствия реализации функциональным требованиям.

Методология тестирования основана на принципе пирамиды тестирования, где наибольшее количество тестов приходится на модульное тестирование, меньшее количество на интеграционное тестирование и наименьшее на end-to-end тестирование. Такой подход обеспечивает баланс между полнотой покрытия, скоростью выполнения тестов и стоимостью поддержки тестового кода.

Модульное тестирование направлено на проверку отдельных функций и методов в изоляции от других компонентов системы. Для серверной части используется фреймворк Jest, который предоставляет возможности для написания и выполнения тестов, создания моков и проверки утверждений. Каждый сервис тестируется отдельно с использованием моков для зависимостей, таких как PrismaService и другие сервисы. Это позволяет проверить логику работы методов без необходимости подключения к реальной базе данных или внешним сервисам.

Интеграционное тестирование проверяет взаимодействие между различными компонентами системы. Для серверной части это включает тестирование контроллеров с реальными сервисами и тестовой базой данных. Используется in-memory база данных SQLite для ускорения выполнения тестов и обеспечения изоляции. Тесты проверяют корректность обработки HTTP-запросов, валидацию входных данных, проверку прав доступа и формирование ответов.

End-to-end тестирование проверяет работу всего приложения от пользовательского интерфейса до базы данных. Для клиентской части может использоваться фреймворк Playwright или Cypress для автоматизации действий пользователя в браузере. Тесты имитируют реальные сценарии использования приложения, такие как регистрация пользователя, создание команды, создание челленджа и отслеживание привычек.

Тестирование валидации форм ввода проверяет корректность обработки пользовательского ввода на клиенте и сервере. На клиенте используется библиотека Zod для определения схем валидации, которые автоматически применяются к формам через React Hook Form. На сервере используются декораторы class-validator для валидации DTO. Тесты проверяют, что некорректные данные отклоняются с соответствующими сообщениями об ошибках, а корректные данные принимаются и обрабатываются правильно.

Тестирование бизнес-логики проверяет корректность реализации бизнес-правил и алгоритмов. Это включает проверку прав доступа, валидацию бизнес-ограничений, расчет статистики и формирование лидербордов. Тесты используют различные сценарии с граничными условиями для обеспечения надежности системы.

Тестирование производительности оценивает скорость работы приложения под различными нагрузками. Используются инструменты для нагрузочного тестирования, такие как Apache JMeter или k6, для симуляции множественных одновременных пользователей. Измеряется время отклика API, пропускная способность и использование ресурсов. Результаты используются для выявления узких мест и оптимизации производительности.

Тестирование безопасности проверяет защищенность приложения от распространенных уязвимостей. Проверяется защита от SQL-инъекций через использование параметризованных запросов Prisma. Проверяется защита от XSS-атак через экранирование пользовательского ввода в React. Проверяется защита от CSRF-атак через использование токенов и проверку origin. Проверяется надежность системы аутентификации и авторизации.

Регрессионное тестирование выполняется после внесения изменений в код для проверки, что новые изменения не нарушили существующую функциональность. Автоматизированные тесты запускаются в continuous integration pipeline при каждом коммите для быстрого выявления проблем. Используется GitHub Actions или GitLab CI для автоматизации процесса тестирования и развертывания.

## 4.2 Валидация форм ввода

### 4.2.1 Тестирование формы регистрации

Форма регистрации является критически важным компонентом системы, обеспечивающим создание новых учетных записей пользователей. Валидация формы регистрации выполняется на двух уровнях: на клиенте для немедленной обратной связи пользователю и на сервере для обеспечения безопасности и целостности данных.

На клиентской стороне схема валидации определена с использованием библиотеки Zod. Схема включает правила для каждого поля формы. Поле email должно быть валидным email адресом, что проверяется через встроенный валидатор email. Поле не может быть пустым. Если пользователь вводит некорректный email, отображается сообщение об ошибке "Введите корректный email адрес".

Поле password должно содержать минимум восемь символов для обеспечения минимального уровня безопасности. Рекомендуется использовать комбинацию букв, цифр и специальных символов, хотя это не является обязательным требованием. Если пользователь вводит пароль короче восьми символов, отображается сообщение "Пароль должен содержать минимум 8 символов". Поле не может быть пустым.

Поле nickname должно содержать от трех до тридцати символов. Никнейм может включать буквы латинского алфавита, цифры, дефисы и подчеркивания. Использование специальных символов и пробелов не допускается для обеспечения корректного отображения в URL и предотвращения проблем с кодировкой. Если пользователь вводит никнейм короче трех символов, отображается сообщение "Никнейм должен содержать минимум 3 символа". Если никнейм длиннее тридцати символов, отображается сообщение "Никнейм не может быть длиннее 30 символов".

Валидация выполняется в режиме реального времени при потере фокуса поля или при попытке отправки формы. Это обеспечивает немедленную обратную связь пользователю без необходимости отправки данных на сервер. Сообщения об ошибках отображаются под соответствующими полями красным цветом для визуального выделения проблемных мест.

На серверной стороне валидация выполняется через DTO RegisterDto с использованием декораторов class-validator. Декораторы дублируют правила валидации клиентской стороны для обеспечения безопасности, так как клиентская валидация может быть обойдена злоумышленником. ValidationPipe автоматически применяет валидацию ко всем входящим запросам и возвращает детальные сообщения об ошибках при невалидных данных.

Дополнительная валидация на сервере проверяет уникальность email и nickname. После прохождения базовой валидации формата AuthService выполняет запросы к базе данных для проверки существования пользователя с таким email или nickname. Если email уже зарегистрирован, возвращается ошибка с сообщением "Пользователь с таким email уже существует". Если nickname уже занят, возвращается ошибка с сообщением "Пользователь с таким никнеймом уже существует". Это предотвращает создание дубликатов и обеспечивает уникальность идентификаторов пользователей.

Тестовые сценарии для формы регистрации включают проверку различных комбинаций входных данных. Позитивный сценарий проверяет успешную регистрацию с корректными данными. Пользователь вводит валидный email, пароль из восьми символов и уникальный никнейм. Форма проходит клиентскую валидацию. Запрос отправляется на сервер. Сервер проверяет уникальность email и nickname. Создается новая учетная запись. Возвращаются данные пользователя и токены. Пользователь автоматически входит в систему и перенаправляется на страницу профиля.

Негативные сценарии проверяют обработку некорректных данных. Сценарий с невалидным email проверяет, что форма не отправляется при вводе email без символа @ или без доменной части. Отображается сообщение об ошибке под полем email. Кнопка отправки остается активной, но клик не приводит к отправке запроса. Сценарий с коротким паролем проверяет, что форма не отправляется при вводе пароля короче восьми символов. Отображается соответствующее сообщение об ошибке.

Сценарий с коротким никнеймом проверяет, что форма не отправляется при вводе никнейма короче трех символов. Сценарий с длинным никнеймом проверяет, что форма не отправляется при вводе никнейма длиннее тридцати символов. Сценарий с дублирующимся email проверяет, что сервер возвращает ошибку при попытке регистрации с email, который уже существует в базе данных. Клиент отображает сообщение об ошибке, полученное от сервера. Сценарий с дублирующимся nickname аналогично проверяет обработку занятого никнейма.

Граничные условия тестируются для проверки корректности обработки крайних значений. Пароль ровно из восьми символов должен проходить валидацию. Никнейм ровно из трех символов должен проходить валидацию. Никнейм ровно из тридцати символов должен проходить валидацию. Эти тесты обеспечивают, что граничные значения обрабатываются корректно без off-by-one ошибок.

### 4.2.2 Тестирование формы создания команды

Форма создания команды позволяет пользователям организовывать новые команды для совместного участия в челленджах. Валидация формы обеспечивает корректность введенных данных и предотвращает создание команд с некорректными параметрами.

На клиентской стороне схема валидации определена для всех полей формы. Поле name является обязательным и должно содержать от трех до пятидесяти символов. Название команды должно быть достаточно описательным, но не слишком длинным для удобного отображения в интерфейсе. Если пользователь пытается создать команду с названием короче трех символов, отображается сообщение "Название должно содержать минимум 3 символа". Если название длиннее пятидесяти символов, отображается сообщение "Название не может быть длиннее 50 символов".

Поле slogan является опциональным и может содержать до ста символов. Слоган представляет краткое описание команды или ее девиз. Если пользователь вводит слоган длиннее ста символов, отображается сообщение "Слоган не может быть длиннее 100 символов". Пустое поле slogan допускается и не вызывает ошибок валидации.

Поле description является опциональным и может содержать до пятисот символов. Описание позволяет детально рассказать о целях команды, правилах участия и других важных аспектах. Если пользователь вводит описание длиннее пятисот символов, отображается сообщение "Описание не может быть длиннее 500 символов". Пустое поле description допускается.

Поле headerBgUrl является опциональным и должно содержать валидный URL при заполнении. Валидация проверяет формат URL с использованием регулярного выражения или встроенного валидатора. Если пользователь вводит некорректный URL, отображается сообщение "Введите корректный URL изображения". Пустое поле допускается, в этом случае используется фон по умолчанию.

Чекбоксы isPrivate, requireInviteCode и requireEmailWhitelist не требуют валидации, так как они могут быть только в состоянии checked или unchecked. Однако комбинация этих флагов влияет на логику присоединения к команде. Если команда приватная, она не отображается в общем списке команд. Если требуется код приглашения, пользователи могут присоединиться только по валидному коду. Если требуется whitelist email, проверяется наличие email пользователя в whitelist при попытке присоединения.

На серверной стороне валидация выполняется через CreateTeamDto. Декораторы проверяют типы данных, обязательность полей и ограничения по длине. Дополнительная бизнес-логика в TeamsService проверяет, что пользователь аутентифицирован и имеет право создавать команды. В текущей реализации нет ограничений на количество команд, которые может создать пользователь, но такое ограничение может быть добавлено в будущем.

Тестовые сценарии для формы создания команды включают различные комбинации параметров. Позитивный сценарий с минимальными данными проверяет создание команды только с обязательным полем name. Пользователь вводит название из трех символов. Опциональные поля остаются пустыми. Чекбоксы остаются в состоянии по умолчанию unchecked. Форма проходит валидацию. Запрос отправляется на сервер. Создается публичная команда без требований к присоединению. Пользователь перенаправляется на страницу созданной команды.

Позитивный сценарий с полными данными проверяет создание команды со всеми заполненными полями. Пользователь вводит название, слоган, описание и URL фонового изображения. Устанавливает чекбоксы в соответствии с требованиями. Форма проходит валидацию. Создается команда с указанными параметрами. Все данные корректно сохраняются в базе данных и отображаются на странице команды.

Негативные сценарии проверяют обработку некорректных данных. Сценарий с пустым названием проверяет, что форма не отправляется при пустом обязательном поле name. Отображается сообщение "Название обязательно для заполнения". Сценарий с коротким названием проверяет, что форма не отправляется при названии короче трех символов. Сценарий с длинным названием проверяет, что форма не отправляется при названии длиннее пятидесяти символов.

Сценарий с длинным слоганом проверяет, что форма не отправляется при слогане длиннее ста символов. Сценарий с длинным описанием проверяет, что форма не отправляется при описании длиннее пятисот символов. Сценарий с некорректным URL проверяет, что форма не отправляется при вводе невалидного URL в поле headerBgUrl. Отображается соответствующее сообщение об ошибке.

Граничные условия тестируются для всех полей с ограничениями по длине. Название ровно из трех символов должно проходить валидацию. Название ровно из пятидесяти символов должно проходить валидацию. Слоган ровно из ста символов должен проходить валидацию. Описание ровно из пятисот символов должно проходить валидацию. Эти тесты обеспечивают корректность обработки граничных значений.

## 4.3 Тестирование бизнес-логики

### 4.3.1 Тестирование системы аутентификации

Система аутентификации является критически важным компонентом, обеспечивающим безопасный доступ к ресурсам приложения. Тестирование системы аутентификации проверяет корректность всех операций, связанных с управлением учетными записями и сессиями пользователей.

Тестирование процесса регистрации проверяет создание новых учетных записей. Позитивный сценарий проверяет успешную регистрацию с уникальными учетными данными. Тест создает пользователя с валидным email, паролем и никнеймом. Проверяется, что пользователь создан в базе данных. Проверяется, что пароль хеширован с использованием Argon2 и не хранится в открытом виде. Проверяется, что возвращаются access token и refresh token. Проверяется, что создана сессия в базе данных с корректными параметрами.

Негативный сценарий с дублирующимся email проверяет, что система отклоняет попытку регистрации с email, который уже существует. Тест пытается создать второго пользователя с тем же email. Проверяется, что выбрасывается ConflictException с сообщением о существующем пользователе. Проверяется, что новая запись не создана в базе данных. Аналогичный сценарий проверяет обработку дублирующегося nickname.

Тестирование процесса входа проверяет аутентификацию существующих пользователей. Позитивный сценарий проверяет успешный вход с корректными учетными данными. Тест создает пользователя в базе данных. Выполняет запрос на вход с email и паролем пользователя. Проверяется, что возвращаются данные пользователя и токены. Проверяется, что создана новая сессия. Проверяется, что токены валидны и содержат корректные данные в payload.

Негативный сценарий с неверным email проверяет, что система отклоняет попытку входа с несуществующим email. Тест выполняет запрос с email, который не зарегистрирован в системе. Проверяется, что выбрасывается UnauthorizedException с общим сообщением о неверных учетных данных. Проверяется, что сессия не создана. Негативный сценарий с неверным паролем проверяет, что система отклоняет попытку входа с некорректным паролем. Тест выполняет запрос с корректным email, но неверным паролем. ## 4.4 Тестирование производительности

Тестирование производительности оценивает скорость работы приложения и его способность обрабатывать множественные одновременные запросы. Целью является выявление узких мест и обеспечение приемлемого времени отклика для пользователей.

Тестирование времени отклика API измеряет скорость обработки различных типов запросов. Для простых запросов на получение данных, таких как получение списка команд или информации о пользователе, целевое время отклика составляет менее двухсот миллисекунд. Тесты выполняют множественные запросы и измеряют среднее, минимальное и максимальное время отклика. Результаты сравниваются с целевыми значениями для выявления проблем производительности.

Для сложных запросов, таких как получение данных таблицы трекинга челленджа с большим количеством привычек и участников, целевое время отклика составляет менее одной секунды. Тесты создают челлендж с десятью привычками и двадцатью участниками за период тридцать дней, что дает шесть тысяч потенциальных логов. Измеряется время выполнения запроса на получение данных таблицы. Анализируются SQL-запросы, выполняемые Prisma, для выявления неоптимальных запросов.

Тестирование пропускной способности измеряет количество запросов, которое сервер может обработать в единицу времени. Используется инструмент для нагрузочного тестирования, такой как Apache JMeter или k6, для симуляции множественных одновременных пользователей. Тесты постепенно увеличивают количество виртуальных пользователей от десяти до ста и измеряют количество успешно обработанных запросов в секунду. Мониторится использование CPU, памяти и сетевых ресурсов сервера.

Целевая пропускная способность составляет минимум сто запросов в секунду для простых операций чтения. Для операций записи, таких как создание логов выполнения привычек, целевая пропускная способность составляет минимум пятьдесят запросов в секунду. Результаты используются для определения необходимости горизонтального масштабирования или оптимизации кода.

Тестирование производительности базы данных оценивает скорость выполнения SQL-запросов. Анализируются медленные запросы через логи Prisma или инструменты мониторинга PostgreSQL. Проверяется наличие индексов на часто используемых полях для фильтрации и сортировки. Тесты выполняют запросы с различными параметрами и измеряют время выполнения.

Для оптимизации производительности применяются следующие техники. Добавление индексов на поля, используемые в WHERE и JOIN клаузах. Использование select для загрузки только необходимых полей вместо всех полей сущности. Использование пагинации для ограничения количества возвращаемых записей. Использование кеширования для часто запрашиваемых данных, которые редко изменяются.

Тестирование производительности клиентской части оценивает скорость загрузки и рендеринга страниц. Используются инструменты Google Lighthouse или WebPageTest для измерения метрик производительности. Измеряется First Contentful Paint, время до первого отображения контента. Измеряется Largest Contentful Paint, время до отображения основного контента. Измеряется Time to Interactive, время до полной интерактивности страницы.

Целевые значения для производительности клиента: FCP менее одной секунды, LCP менее двух с половиной секунд, TTI менее трех с половиной секунд. Для достижения этих значений применяются техники оптимизации. Использование серверного рендеринга Next.js для ускорения первой загрузки. Ленивая загрузка компонентов и изображений. Минификация и сжатие JavaScript и CSS файлов. Использование CDN для статических ресурсов.

Тестирование производительности 3D визуализации оценивает частоту кадров при рендеринге MountainLeaderboard компонента. Целевая частота составляет минимум тридцать кадров в секунду для плавной анимации. Тесты измеряют FPS при различном количестве участников на лидерборде. Проверяется производительность на различных устройствах, включая мобильные с менее мощными GPU.

Для оптимизации 3D производительности применяются техники упрощения геометрии для уменьшения количества полигонов. Использование level of detail для отображения упрощенных моделей на расстоянии. Ограничение количества источников света в сцене. Использование текстур с разумным разрешением.
