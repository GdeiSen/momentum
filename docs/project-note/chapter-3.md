# 3 Разработка веб-приложения

## 3.1 Разработка серверной части

### 3.1.1 Структура серверной части приложения

Серверная часть приложения Momentum реализована с использованием фреймворка NestJS версии 10.4.15, который представляет собой прогрессивный Node.js фреймворк для построения эффективных и масштабируемых серверных приложений. NestJS построен на основе TypeScript и использует архитектурные паттерны, заимствованные из Angular, что обеспечивает структурированный и предсказуемый подход к разработке.

Корневая директория серверного приложения организована следующим образом. Директория src содержит весь исходный код приложения. Директория prisma содержит схему базы данных и миграции. Файл package.json определяет зависимости проекта и скрипты для запуска и сборки приложения. Файлы конфигурации TypeScript tsconfig.json и tsconfig.build.json определяют параметры компиляции. Файлы Docker Dockerfile и Dockerfile.dev определяют образы для продакшена и разработки соответственно.

Внутри директории src код организован по модульному принципу. Файл main.ts является точкой входа в приложение и выполняет инициализацию NestJS приложения с настройкой глобальных параметров. Файл app.module.ts определяет корневой модуль, который импортирует все функциональные модули. Директория common содержит общие компоненты, используемые в различных модулях, включая декораторы, guards и типы. Директория modules содержит функциональные модули приложения, каждый из которых инкапсулирует связанную функциональность. Директория prisma содержит модуль для работы с базой данных через Prisma ORM.

Каждый функциональный модуль имеет единую структуру. Файл с суффиксом module.ts определяет модуль с его зависимостями, контроллерами и провайдерами. Файл с суффиксом controller.ts содержит контроллер, обрабатывающий HTTP-запросы. Файл с суффиксом service.ts содержит сервис с бизнес-логикой. Директория dto содержит объекты передачи данных для валидации входящих данных. Некоторые модули могут содержать дополнительные файлы, такие как guards для проверки прав доступа или strategies для аутентификации.

Система зависимостей управляется через npm, и все зависимости определены в файле package.json. Основные зависимости включают nestjs/common, nestjs/core и nestjs/platform-express для базовой функциональности NestJS. Для работы с базой данных используются prisma/client и nestjs/prisma. Для аутентификации используются nestjs/jwt, nestjs/passport, passport, passport-jwt и argon2 для хеширования паролей. Для валидации данных используются class-validator и class-transformer. Для работы с файлами используется minio. Для генерации документации API используются nestjs/swagger и swagger-ui-express.

Конфигурация приложения управляется через переменные окружения, которые загружаются из файла .env с использованием библиотеки dotenv. Модуль ConfigModule из nestjs/config обеспечивает типобезопасный доступ к переменным окружения через ConfigService. Основные переменные окружения включают DATABASE_URL для подключения к PostgreSQL, JWT_ACCESS_SECRET и JWT_REFRESH_SECRET для подписи токенов, JWT_ACCESS_EXPIRATION и JWT_REFRESH_EXPIRATION для определения времени жизни токенов, параметры подключения к MinIO и PORT для определения порта сервера.

Система логирования реализована через встроенный Logger из nestjs/common. Логи выводятся в консоль с цветовой кодировкой для различных уровней важности. В режиме разработки Prisma настроен на логирование всех SQL-запросов для отладки. В продакшене логируются только ошибки и важные события. Структурированное логирование обеспечивает возможность анализа логов и мониторинга работы приложения.

Обработка ошибок реализована через систему исключений NestJS. Все ошибки бизнес-логики выбрасываются как HTTP-исключения с соответствующими кодами состояния. NotFoundException используется для ошибок, когда запрошенный ресурс не найден. ForbiddenException используется для ошибок прав доступа. ConflictException используется для ошибок дублирования данных. BadRequestException используется для ошибок валидации. UnauthorizedException используется для ошибок аутентификации. Глобальный фильтр исключений перехватывает все необработанные ошибки и формирует единообразные ответы.

### 3.1.2 Модуль аутентификации и авторизации

Модуль аутентификации AuthModule является критически важным компонентом системы, обеспечивающим безопасный доступ к ресурсам приложения. Модуль реализует аутентификацию на основе JWT токенов с механизмом автоматического обновления и управлением сессиями.

AuthModule импортирует PassportModule для интеграции с библиотекой Passport.js и JwtModule для работы с JWT токенами. Конфигурация JwtModule выполняется асинхронно с использованием ConfigService для получения секретного ключа и времени жизни токена из переменных окружения. Модуль объявляет AuthController для обработки HTTP-запросов и несколько сервисов: AuthService для основной логики аутентификации, SessionsService для управления сессиями, JwtStrategy и JwtRefreshStrategy для валидации токенов. Модуль экспортирует AuthService и SessionsService для использования в других модулях.

AuthService содержит методы для всех операций аутентификации. Метод register принимает данные регистрации, включая email, пароль и никнейм. Сервис проверяет уникальность email и nickname, выполняя запросы к базе данных. Если email или nickname уже заняты, выбрасывается ConflictException с соответствующим сообщением. Пароль хешируется с использованием библиотеки Argon2 с параметрами, включающими memory cost равный шестьдесят пять тысяч пятьсот тридцать шесть килобайт, time cost равный три итерации и parallelism равный четыре потока. Эти параметры обеспечивают баланс между безопасностью и производительностью согласно рекомендациям OWASP.

После хеширования пароля создается запись пользователя в базе данных через Prisma Client. Затем вызывается приватный метод generateTokensAndCreateSession для генерации пары токенов и создания сессии. Метод возвращает объект с данными пользователя, access token, refresh token и временем жизни access token. Данные пользователя не включают чувствительные поля, такие как password_hash, для обеспечения безопасности.

Метод login принимает email и пароль для аутентификации существующего пользователя. Сервис выполняет поиск пользователя по email в базе данных. Если пользователь не найден, выбрасывается UnauthorizedException с общим сообщением о неверных учетных данных. Использование общего сообщения предотвращает перечисление пользователей через анализ различий в ответах. Если пользователь найден, извлекается хеш пароля и сравнивается с введенным паролем с использованием функции verify из библиотеки Argon2. При несовпадении паролей выбрасывается то же UnauthorizedException. При успешной проверке генерируются токены и создается сессия, после чего возвращается объект с данными пользователя и токенами.

Метод refreshTokens обрабатывает обновление access token с использованием refresh token. Метод принимает refresh token, опциональный User-Agent и IP-адрес. Сначала токен верифицируется с использованием секретного ключа для refresh токенов. Если токен невалиден или истек, выбрасывается UnauthorizedException. Из payload токена извлекается идентификатор сессии. Выполняется поиск сессии в базе данных через SessionsService. Если сессия не найдена или истекла, выбрасывается UnauthorizedException. Сравнивается refresh token из запроса с токеном, сохраненным в сессии. Несовпадение токенов может указывать на компрометацию и приводит к выбросу UnauthorizedException.

При успешной валидации старая сессия удаляется из базы данных, реализуя ротацию токенов. Создается новая сессия с новой парой токенов. Новые токены возвращаются клиенту. Такой подход обеспечивает безопасность за счет ограничения времени жизни токенов и предотвращения повторного использования refresh токенов. Если refresh token был украден, он может быть использован только один раз, после чего становится невалидным.

Метод logout обрабатывает выход из системы. Метод принимает refresh token и извлекает из него идентификатор сессии. Сессия удаляется из базы данных через SessionsService. Метод возвращает сообщение об успешном выходе. Метод logoutAll обрабатывает выход из всех устройств. Метод принимает идентификатор пользователя и удаляет все его сессии через SessionsService. Это полезно в случае компрометации учетных данных или при необходимости принудительного выхода со всех устройств.

Приватный метод generateTokensAndCreateSession инкапсулирует логику генерации токенов и создания сессии. Метод сначала создает запись сессии в базе данных с временным refresh token для получения идентификатора сессии. Затем генерируется access token с payload, содержащим идентификатор пользователя, email и nickname. Срок действия устанавливается из переменной окружения JWT_ACCESS_EXPIRATION. Токен подписывается секретным ключом JWT_ACCESS_SECRET с использованием алгоритма HS256. Аналогично генерируется refresh token с payload, содержащим идентификатор пользователя и идентификатор сессии. Срок действия устанавливается из переменной окружения JWT_REFRESH_EXPIRATION. Сгенерированный refresh token сохраняется в записи сессии через обновление. Оба токена возвращаются вместе с временем жизни access token.

SessionsService управляет сессиями пользователей в базе данных. Метод create создает новую сессию с указанными параметрами. Метод findById находит сессию по идентификатору. Метод findByIdAndToken находит сессию по идентификатору и токену для валидации. Метод findAllByUserId возвращает все активные сессии пользователя для отображения в настройках. Метод updateRefreshToken обновляет refresh token в сессии. Метод delete удаляет сессию с обработкой случая, когда сессия не найдена. Метод deleteAllByUserId удаляет все сессии пользователя. Метод deleteExpired удаляет истекшие сессии для периодической очистки.

JwtStrategy реализует стратегию валидации access token для Passport.js. Стратегия настроена на извлечение токена из заголовка Authorization с префиксом Bearer. Секретный ключ для верификации берется из конфигурации. Метод validate вызывается автоматически после успешной верификации токена и принимает payload токена. Метод вызывает AuthService для получения данных пользователя и возвращает их. Возвращенные данные прикрепляются к объекту request как свойство user, что позволяет контроллерам получать информацию о текущем пользователе.

JwtRefreshStrategy реализует стратегию валидации refresh token. Стратегия настроена на извлечение токена из поля refreshToken в теле запроса. Секретный ключ для верификации берется из конфигурации для refresh токенов. Метод validate возвращает объект с идентификатором пользователя и идентификатором сессии из payload токена. Эта стратегия используется только для эндпоинта обновления токенов.

AuthController обрабатывает HTTP-запросы, связанные с аутентификацией. Контроллер объявлен с базовым путем auth. Метод register обрабатывает POST-запросы на путь /api/auth/register. Метод аннотирован декоратором Public, что исключает его из проверки аутентификации. Метод принимает RegisterDto из тела запроса и опциональные заголовки User-Agent и IP-адрес. Метод вызывает AuthService.register и возвращает результат с кодом 201 Created.

Метод login обрабатывает POST-запросы на путь /api/auth/login. Метод также аннотирован декоратором Public. Метод принимает LoginDto из тела запроса и опциональные заголовки. Метод вызывает AuthService.login и возвращает результат с кодом 200 OK. Метод refresh обрабатывает POST-запросы на путь /api/auth/refresh. Метод принимает RefreshTokenDto из тела запроса. Метод вызывает AuthService.refreshTokens и возвращает новые токены.

Метод logout обрабатывает POST-запросы на путь /api/auth/logout. Метод принимает RefreshTokenDto из тела запроса. Метод вызывает AuthService.logout и возвращает сообщение об успешном выходе. Метод logoutAll обрабатывает POST-запросы на путь /api/auth/logout-all. Метод защищен JwtAuthGuard и использует декоратор CurrentUser для получения идентификатора пользователя. Метод вызывает AuthService.logoutAll и возвращает сообщение.

Метод getSessions обрабатывает GET-запросы на путь /api/auth/sessions. Метод защищен JwtAuthGuard и возвращает список всех активных сессий пользователя. Метод revokeSession обрабатывает DELETE-запросы на путь /api/auth/sessions/:sessionId. Метод позволяет пользователю отозвать конкретную сессию, например, при выходе с определенного устройства. Метод getMe обрабатывает GET-запросы на путь /api/auth/me. Метод возвращает данные текущего аутентифицированного пользователя для проверки валидности токена на клиенте.

DTO для модуля аутентификации определяют структуру входных данных и правила валидации. RegisterDto содержит поля email, password и nickname. Поле email аннотировано декораторами IsEmail для проверки формата и IsNotEmpty для обязательности. Поле password аннотировано декораторами IsString, IsNotEmpty и MinLength с минимальной длиной восемь символов. Поле nickname аннотировано декораторами IsString, IsNotEmpty, MinLength с минимальной длиной три символа и MaxLength с максимальной длиной тридцать символов.

LoginDto содержит поля email и password с аналогичными декораторами валидации. RefreshTokenDto содержит единственное поле refreshToken, аннотированное декораторами IsString и IsNotEmpty. Все DTO экспортируются из файла index.ts в директории dto для удобного импорта.

### 3.1.3 Модуль управления командами

Модуль управления командами TeamsModule обеспечивает полный жизненный цикл команд, включая создание, редактирование, удаление, управление участниками и настройку параметров доступа. Модуль является одним из наиболее сложных в приложении из-за множества взаимосвязанных сущностей и бизнес-правил.

TeamsModule импортирует PrismaModule для доступа к базе данных. Модуль объявляет TeamsController для обработки HTTP-запросов и несколько сервисов: TeamsService для основной логики работы с командами, TeamMembersService для управления участниками, TeamInvitesService для работы с кодами приглашений и TeamWhitelistService для управления whitelist email. Все сервисы экспортируются для использования в других модулях, так как многие операции требуют проверки членства в команде.

TeamsService содержит методы для всех операций с командами. Метод create принимает идентификатор пользователя и CreateTeamDto с данными команды. Сервис создает команду в базе данных с указанным владельцем. Автоматически создается запись в team_members, связывающая владельца с командой с ролью OWNER. Автоматически создается чат для команды через создание записи в таблице chats. Метод возвращает созданную команду с полной информацией, включая данные владельца.

Метод findAll возвращает список команд с опциональной фильтрацией. Если указан идентификатор пользователя и флаг includePrivate установлен в true, метод возвращает все команды, где пользователь является членом, включая приватные. Иначе возвращаются только публичные команды. Это используется на странице поиска команд, где показываются только публичные команды, доступные для присоединения.

Метод findUserTeams возвращает все команды, где указанный пользователь является членом. Метод выполняет запрос к таблице team_members для получения всех членств пользователя, затем загружает полные данные команд с информацией о владельце и количестве участников. Метод findById возвращает команду по идентификатору с полной информацией. Если команда приватная и пользователь не является членом, выбрасывается ForbiddenException.

Метод update обновляет настройки команды. Метод сначала проверяет, что пользователь является администратором или владельцем команды через TeamMembersService. Если пользователь не имеет необходимых прав, выбрасывается ForbiddenException. Метод обновляет указанные поля команды через Prisma Client. Некоторые поля, такие как is_private, require_invite_code и require_email_whitelist, могут изменять только владельцы команды, что проверяется дополнительно.

Метод delete удаляет команду. Метод проверяет, что пользователь является владельцем команды. Только владелец может удалить команду. Метод удаляет команду из базы данных, что приводит к каскадному удалению всех связанных данных, включая членства, челленджи, привычки, логи, посты, сообщения, коды приглашений и whitelist записи. Перед удалением рекомендуется показывать пользователю предупреждение о необратимости операции.

Метод join обрабатывает присоединение пользователя к публичной команде. Метод проверяет, что команда не является приватной. Метод проверяет, что не требуется код приглашения. Если требуется whitelist email, метод проверяет наличие email пользователя в whitelist через TeamWhitelistService. Если все проверки пройдены, создается запись в team_members с ролью MEMBER. Если пользователь уже является членом команды, выбрасывается ConflictException.

Метод leave обрабатывает выход пользователя из команды. Метод проверяет, что пользователь не является владельцем команды. Владелец не может покинуть команду, он должен либо передать владение другому участнику, либо удалить команду. Метод удаляет запись из team_members, что прекращает доступ пользователя ко всем ресурсам команды.

Метод getStatistics возвращает статистику команды. Метод подсчитывает количество членов через запрос к team_members. ### 3.1.5 Модуль хранения файлов

Модуль хранения файлов StorageModule обеспечивает загрузку, хранение и получение файлов через интеграцию с объектным хранилищем MinIO. MinIO представляет собой S3-совместимое хранилище, что обеспечивает возможность миграции на Amazon S3 или другие S3-совместимые сервисы при необходимости.

StorageModule импортирует ConfigModule для доступа к параметрам подключения к MinIO. Модуль объявляет StorageController и StorageService. Сервис экспортируется для использования в других модулях, так как многие операции требуют загрузки файлов.

StorageService инициализирует клиент MinIO в конструкторе. Параметры подключения, включая endpoint, port, useSSL, accessKey и secretKey, извлекаются из ConfigService. Имя bucket также извлекается из конфигурации. Метод onModuleInit выполняется при инициализации модуля и проверяет существование bucket. Если bucket не существует, метод создает его. Метод устанавливает политику публичного чтения для bucket, что позволяет получать файлы по прямым URL без аутентификации.

Метод uploadFile обрабатывает загрузку файла. Метод принимает буфер файла, оригинальное имя, MIME-тип и папку для организации файлов. Метод валидирует MIME-тип, проверяя, что он соответствует одному из разрешенных типов изображений. Разрешенные типы включают image/jpeg, image/png, image/gif и image/webp. Метод валидирует размер файла, проверяя, что он не превышает пять мегабайт. Если валидация не пройдена, выбрасывается BadRequestException.

Метод генерирует уникальное имя файла, используя UUID и расширение из оригинального имени. Имя файла формируется как folder/uuid.extension. Метод загружает файл в MinIO через метод putObject клиента. Метод формирует публичный URL файла, комбинируя endpoint, порт, имя bucket и имя файла. Метод возвращает URL, который может быть сохранен в базе данных и использован для доступа к файлу.

Метод getPresignedUploadUrl генерирует presigned URL для прямой загрузки с клиента. Метод принимает имя файла, папку и опциональное время жизни URL в секундах. Метод генерирует уникальное имя файла аналогично uploadFile. Метод вызывает метод presignedPutObject клиента MinIO для генерации временного URL с правами на запись. Метод формирует публичный URL файла после загрузки. Метод возвращает объект с uploadUrl для загрузки и fileUrl для последующего доступа.

Использование presigned URLs оптимизирует процесс загрузки файлов. Клиент запрашивает presigned URL у сервера. Сервер генерирует URL и возвращает его клиенту. Клиент загружает файл напрямую в MinIO, используя presigned URL. После успешной загрузки клиент сохраняет fileUrl в базе данных через обычный API запрос. Такой подход снижает нагрузку на сервер приложения и ускоряет загрузку для пользователей.

Метод deleteFile удаляет файл из хранилища. Метод принимает URL файла, извлекает имя объекта из URL и вызывает метод removeObject клиента MinIO. Метод используется при удалении или замене аватаров, обложек и постеров для освобождения пространства. Метод getPresignedDownloadUrl генерирует presigned URL для скачивания файла. Метод может использоваться для приватных файлов, требующих аутентификации для доступа.

StorageController обрабатывает HTTP-запросы для работы с файлами. Метод upload обрабатывает POST-запросы на путь /api/storage/upload. Метод использует декоратор UseInterceptors с FileInterceptor для обработки multipart/form-data. Метод принимает файл через декоратор UploadedFile. Метод принимает folder из тела запроса для организации файлов. Метод вызывает StorageService.uploadFile и возвращает URL загруженного файла.

Метод getPresignedUrl обрабатывает POST-запросы на путь /api/storage/presigned-url. Метод принимает GetPresignedUrlDto из тела запроса. DTO содержит поля fileName и folder. Метод вызывает StorageService.getPresignedUploadUrl и возвращает объект с uploadUrl и fileUrl. Метод deleteFile обрабатывает DELETE-запросы на путь /api/storage. Метод принимает fileUrl из query параметров. Метод вызывает StorageService.deleteFile и возвращает сообщение об успешном удалении.

Организация файлов в хранилище выполнена по папкам в соответствии с типом контента. Папка avatars содержит аватары пользователей. Папка headers содержит фоновые изображения профилей и команд. Папка posters содержит постеры челленджей. Папка chat-media содержит медиафайлы из сообщений чата. Папка post-media содержит медиафайлы из постов. Такая организация упрощает управление файлами и настройку политик доступа.

## 3.2 Разработка клиентской части

### 3.2.1 Структура клиентской части приложения

Клиентская часть приложения Momentum разработана с использованием фреймворка Next.js версии 16.0.5, который обеспечивает серверный рендеринг, оптимизацию производительности и удобную систему маршрутизации. Приложение использует React версии 19.2.0 для построения пользовательского интерфейса и TypeScript версии 5 для типобезопасности.

Корневая директория клиентского приложения организована следующим образом. Директория app содержит страницы и маршруты приложения с использованием App Router. Директория components содержит переиспользуемые React компоненты. Директория lib содержит утилиты, API клиенты и stores для управления состоянием. Директория types содержит глобальные типы TypeScript. Директория public содержит статические файлы, такие как изображения и иконки. Файл package.json определяет зависимости проекта. Файлы конфигурации TypeScript и Next.js определяют параметры компиляции и сборки.

Структура директории app организована по принципу файловой маршрутизации Next.js. Группа маршрутов auth содержит страницы аутентификации login и register. Группа маршрутов dashboard содержит защищенные страницы приложения, включая profile, teams, challenges и settings. Динамические маршруты используют квадратные скобки для параметров. Например, маршрут teams/[teamId] обрабатывает страницы команд, где teamId является параметром. Вложенный маршрут teams/[teamId]/challenges/[challengeId] обрабатывает страницы челленджей.

Каждый маршрут содержит файл page.tsx, который экспортирует компонент страницы. Компонент может быть серверным или клиентским в зависимости от требований. Серверные компоненты используются для статического контента и улучшают производительность за счет рендеринга на сервере. Клиентские компоненты используются для интерактивных элементов и помечаются директивой use client в начале файла.

Файл layout.tsx в корне app определяет общий макет для всех страниц. Макет включает HTML структуру, подключение шрифтов, провайдеры контекста и глобальные стили. Вложенные layout.tsx в группах маршрутов определяют дополнительные макеты для конкретных разделов. Например, layout.tsx в группе dashboard включает Header и Sidebar компоненты, которые отображаются на всех страницах дашборда.

Директория components организована по функциональным областям. Поддиректория ui содержит базовые компоненты из библиотеки shadcn/ui. Эти компоненты построены на основе Radix UI и стилизованы с помощью Tailwind CSS. Компоненты включают Button, Card, Dialog, Input, Select, Tabs и многие другие. Все компоненты полностью типизированы и поддерживают кастомизацию через props.

Поддиректория layout содержит компоненты макета. Header отображает шапку приложения с логотипом, навигационным меню и аватаром пользователя. Sidebar отображает боковую панель с навигационными ссылками. Hero отображает 3D фон с анимированными объектами. ContentWrapper обеспечивает правильные отступы и центрирование контента.

Поддиректории challenge, team, workspace и mountain содержат специализированные компоненты для соответствующих функциональных областей. Компоненты challenge включают HabitTrackerTable для отображения таблицы трекинга, PosterCircleBlock и PosterRailBlock для визуализации челленджей. Компоненты team включают PostList, CreatePostForm, TeamChat, ChartBlock и RadarChartBlock. Компоненты workspace включают DesktopWorkspace и различные виджеты. Компоненты mountain включают MountainLeaderboard для 3D визуализации лидерборда.

Директория lib содержит утилиты и библиотеки. Поддиректория api содержит API клиенты для взаимодействия с сервером. Файл client.ts предоставляет базовую функцию apiRequest. Специализированные файлы, такие как auth.ts, teams.ts, challenges.ts, предоставляют методы для конкретных эндпоинтов. Поддиректория stores содержит Zustand stores для управления состоянием. Файл utils.ts содержит вспомогательные функции, такие как cn для объединения классов Tailwind.

Система зависимостей управляется через npm. Основные зависимости включают next, react и react-dom для базовой функциональности. Для управления состоянием используется zustand. Для работы с формами используются react-hook-form и zod. Для стилизации используются tailwindcss и библиотека компонентов shadcn/ui. Для работы с датами используется date-fns. Для уведомлений используется sonner. Для 3D графики используются three и react-three-fiber. Для графиков используются recharts и chart.js.

### 3.2.2 Система управления состоянием

Система управления состоянием реализована через библиотеку Zustand, которая предоставляет простой и эффективный способ работы с глобальным состоянием без избыточной сложности. Zustand использует хуки React для подписки компонентов на изменения состояния и автоматически перерисовывает только те компоненты, которые используют измененные данные.

AuthStore управляет состоянием аутентификации и является наиболее критичным store в приложении. Store определен в файле lib/stores/auth-store.ts. Интерфейс AuthState определяет структуру состояния, включая поля user, accessToken, refreshToken, isAuthenticated, isLoading и isHydrated. Интерфейс AuthActions определяет методы для изменения состояния, включая setAuth, setUser, logout, setLoading и setHydrated.

Store создается с использованием функции create из zustand. Начальное состояние устанавливает user в null, токены в пустые строки, isAuthenticated в false, isLoading в true и isHydrated в false. Метод setAuth принимает данные пользователя и токены, устанавливает их в состояние и устанавливает isAuthenticated в true. Метод setUser обновляет только данные пользователя без изменения токенов. Метод logout очищает все данные.

Store использует middleware persist для автоматического сохранения состояния в localStorage. Конфигурация persist определяет имя ключа для хранения как momentum-auth. Определяется callback onRehydrateStorage, который вызывается после восстановления состояния из localStorage. Callback устанавливает isHydrated в true и isLoading в false.

ThemeStore управляет настройками темы и фона. Store определен в файле lib/stores/theme-store.ts. Состояние включает поля theme, backgroundMode, backgroundShape и backgroundImage. Методы setTheme, setBackgroundMode, setBackgroundShape и setBackgroundImage обновляют соответствующие поля. Store также использует middleware persist для сохранения настроек в localStorage с ключом momentum-theme.

UIStore управляет временным состоянием пользовательского интерфейса. Store определен в файле lib/stores/ui-store.ts. Состояние включает поля для конфигурации камеры 3D сцены, кастомных фоновых изображений для Hero компонента и других UI-специфичных настроек. Store не использует persist, так как состояние не должно сохраняться между сессиями.

### 3.2.3 Взаимодействие с серверным API

Взаимодействие с серверным API организовано через модульную систему API клиентов. Базовый модуль client.ts предоставляет функцию apiRequest, которая является оберткой над fetch API с дополнительной функциональностью.

Функция apiRequest принимает endpoint и опциональный объект options. Endpoint представляет путь относительно базового URL API. Options включает method, body, headers и флаг skipAuth. Функция формирует полный URL, комбинируя базовый URL из переменной окружения NEXT_PUBLIC_API_URL и endpoint.

Функция извлекает access token из AuthStore, если skipAuth не установлен в true. Токен добавляется в заголовок Authorization с префиксом Bearer. Если body предоставлен, он сериализуется в JSON и добавляется заголовок Content-Type application/json. Функция выполняет fetch запрос с сформированными параметрами.

После получения ответа функция проверяет статус код. Если статус 401 Unauthorized и skipAuth не установлен, функция пытается обновить токен через функцию refreshAccessToken. Если обновление успешно, новые токены сохраняются в AuthStore. Исходный запрос повторяется с новым access token.

Специализированные API клиенты используют функцию apiRequest для выполнения запросов к конкретным эндпоинтам. Каждый клиент экспортирует объект с методами для различных операций. Все методы типизированы с использованием TypeScript интерфейсов.

## 3.3 Контейнеризация приложения с использованием Docker

### 3.3.1 Структура Docker-контейнеров

Контейнеризация приложения выполнена с использованием Docker для обеспечения консистентности окружения разработки, тестирования и продакшена. Каждый компонент системы упакован в отдельный Docker-контейнер с четко определенными зависимостями и конфигурацией.

Серверное приложение упаковано в контейнер на основе образа node:20-alpine. Alpine Linux выбран для минимизации размера образа и улучшения безопасности. Dockerfile использует многоэтапную сборку для оптимизации размера финального образа. Первый этап выполняет установку зависимостей и сборку TypeScript кода. Второй этап копирует только необходимые файлы для запуска приложения.

Клиентское приложение упаковано в контейнер с аналогичным подходом. Dockerfile использует многоэтапную сборку. Первый этап выполняет установку зависимостей и сборку Next.js приложения. Второй этап копирует только необходимые файлы для запуска Next.js сервера в production режиме.

База данных PostgreSQL использует официальный образ postgres:16-alpine. Образ настраивается через переменные окружения. Данные базы сохраняются в Docker volume для персистентности. Объектное хранилище MinIO использует официальный образ minio/minio. Данные MinIO также сохраняются в Docker volume.

### 3.3.2 Конфигурация Docker Compose

Оркестрация всех контейнеров выполнена с использованием Docker Compose. Файл docker-compose.yml определяет конфигурацию всех сервисов, их взаимосвязи, сетевые настройки и параметры запуска.

Сервис postgres определяет контейнер базы данных. Используется образ postgres:16-alpine. Переменные окружения устанавливают параметры базы данных. Volume postgres-data монтируется для персистентности данных. Порт 5432 пробрасывается на хост. Healthcheck определяет команду для проверки готовности базы данных.

Сервис minio определяет контейнер объектного хранилища. Используется образ minio/minio. Переменные окружения устанавливают учетные данные. Volume minio-data монтируется для персистентности. Порты 9000 и 9001 пробрасываются для API и консоли.

Сервис server определяет контейнер серверного приложения. Используется build контекст для сборки образа. Переменные окружения включают DATABASE_URL, параметры JWT и параметры MinIO. Зависимость depends_on указывает на postgres и minio. Порт 3001 пробрасывается на хост.

Сервис client определяет контейнер клиентского приложения. Используется build контекст для сборки образа. Переменная окружения NEXT_PUBLIC_API_URL указывает на URL серверного API. Зависимость depends_on указывает на server. Порт 3000 пробрасывается на хост.

Volumes определяют именованные volumes для персистентности данных. Networks определяет пользовательскую сеть для связи между контейнерами. Все сервисы подключены к сети momentum-network.

Запуск приложения выполняется командой docker-compose up. Команда собирает образы, создает контейнеры, создает сеть и volumes, запускает контейнеры в правильном порядке. Остановка приложения выполняется командой docker-compose down.

## 3.4 Выводы

В данной главе была описана разработка веб-приложения Momentum, включающая реализацию серверной части на базе NestJS, клиентской части на базе Next.js и контейнеризацию всех компонентов с использованием Docker.

Серверная часть реализована с использованием модульной архитектуры NestJS, где каждый модуль инкапсулирует связанную функциональность. Модуль аутентификации обеспечивает безопасный доступ к ресурсам через JWT токены с механизмом автоматического обновления. Модуль управления командами реализует полный жизненный цикл команд с гибкой системой ролей и параметров доступа. Модули управления челленджами и привычками обеспечивают основной функционал трекинга привычек с детальной валидацией и проверкой прав доступа. Модуль хранения файлов интегрирован с MinIO для эффективного управления медиафайлами.

Клиентская часть разработана с использованием современных технологий и паттернов. Компонентный подход обеспечивает переиспользуемость и поддерживаемость кода. Система управления состоянием через Zustand обеспечивает централизованное хранение данных с персистентностью в localStorage. Ключевые компоненты реализуют сложную функциональность с оптимизацией производительности. Взаимодействие с серверным API организовано через модульную систему API клиентов с единообразной обработкой ошибок и автоматическим обновлением токенов.

Контейнеризация приложения с использованием Docker обеспечивает консистентность окружения и упрощает развертывание. Многоэтапная сборка Docker образов оптимизирует размер финальных образов. Docker Compose оркестрирует все компоненты системы, определяя их взаимосвязи и параметры запуска. Использование volumes обеспечивает персистентность данных между перезапусками контейнеров.

Разработанное приложение реализует все функциональные требования, определенные в первой главе. Модульная архитектура обеспечивает возможность дальнейшего расширения функционала. Использование современных технологий и лучших практик разработки обеспечивает высокую производительность, безопасность и удобство использования системы.
