# 2 Разработка архитектуры проекта

## 2.1 Структура управления приложения

Архитектура веб-приложения Momentum построена на принципах разделения ответственности и модульности, что обеспечивает высокую поддерживаемость, масштабируемость и возможность параллельной разработки различных компонентов системы. Приложение реализовано по архитектурному паттерну клиент-сервер, где клиентская часть отвечает за представление данных и взаимодействие с пользователем, а серверная часть обеспечивает бизнес-логику, хранение данных и управление доступом.

Серверная часть приложения построена на основе фреймворка NestJS, который реализует модульную архитектуру с использованием паттерна внедрения зависимостей. Каждый функциональный модуль инкапсулирует связанную функциональность и может быть разработан, протестирован и развернут независимо от других модулей. Модули взаимодействуют между собой через четко определенные интерфейсы, что минимизирует связанность компонентов и упрощает внесение изменений. Основными модулями серверной части являются модуль аутентификации, модуль управления пользователями, модуль управления командами, модуль управления челленджами, модуль управления привычками, модуль логирования выполнения привычек, модуль постов, модуль чатов, модуль хранения файлов и модуль пользовательских настроек.

Клиентская часть приложения разработана с использованием фреймворка Next.js, который обеспечивает серверный рендеринг страниц для улучшения производительности и поисковой оптимизации. Архитектура клиентской части организована по принципу компонентного подхода, где каждый компонент представляет собой независимую единицу пользовательского интерфейса с собственной логикой и состоянием. Компоненты организованы иерархически, где более сложные компоненты композируются из простых базовых компонентов. Управление глобальным состоянием приложения осуществляется через библиотеку Zustand, которая обеспечивает централизованное хранилище данных с возможностью подписки компонентов на изменения.

Взаимодействие между клиентской и серверной частями осуществляется через RESTful API, построенное в соответствии с принципами REST архитектуры. API предоставляет набор эндпоинтов для выполнения операций создания, чтения, обновления и удаления данных. Все запросы к API защищены системой аутентификации на основе JWT токенов, которые передаются в заголовке Authorization каждого запроса. Система автоматического обновления токенов обеспечивает непрерывную работу пользователя без необходимости повторной авторизации при истечении срока действия access token.

Система управления доступом реализована на основе ролевой модели с тремя основными ролями в контексте команды. Роль владельца команды предоставляет полный контроль над командой, включая возможность удаления команды, изменения всех настроек, управления участниками и назначения администраторов. Роль администратора команды обеспечивает расширенные права по управлению контентом команды, включая создание и редактирование челленджей, управление привычками и модерацию активности участников. Роль обычного участника команды предоставляет базовые права на просмотр информации команды, участие в челленджах и отслеживание собственных привычек. Система проверки прав доступа реализована через механизм Guards в NestJS, которые выполняют валидацию прав пользователя перед выполнением запрошенной операции.

Архитектура базы данных построена на основе реляционной модели с использованием PostgreSQL в качестве СУБД. Для взаимодействия с базой данных используется ORM Prisma, который обеспечивает типобезопасный доступ к данным и автоматическую генерацию SQL-запросов на основе объектной модели. Prisma также предоставляет механизм миграций для управления схемой базы данных и обеспечения консистентности структуры данных между различными окружениями. Схема базы данных спроектирована с учетом нормализации для минимизации избыточности данных и обеспечения целостности информации.

Система хранения файлов организована с использованием объектного хранилища MinIO, которое обеспечивает S3-совместимый API для загрузки, хранения и получения файлов. Файлы организованы в бакеты по типам контента, что упрощает управление и настройку политик доступа. Для оптимизации производительности используется механизм presigned URLs, который позволяет клиентскому приложению загружать файлы напрямую в хранилище, минуя сервер приложения. Это снижает нагрузку на сервер и ускоряет процесс загрузки файлов для пользователей.

Контейнеризация всех компонентов приложения выполнена с использованием Docker, что обеспечивает консистентность окружения разработки, тестирования и продакшена. Каждый компонент системы упакован в отдельный Docker-контейнер с четко определенными зависимостями и конфигурацией. Оркестрация контейнеров осуществляется с помощью Docker Compose, который определяет конфигурацию всех сервисов, их взаимосвязи, сетевые настройки и параметры запуска. Такой подход упрощает развертывание приложения на различных платформах и обеспечивает возможность масштабирования отдельных компонентов независимо друг от друга.

## 2.2 Диаграммы UML и взаимосвязь компонентов

### 2.2.1 Диаграмма вариантов использования

Диаграмма вариантов использования описывает функциональные требования к системе с точки зрения внешних пользователей и определяет основные сценарии взаимодействия различных ролей с приложением. В системе Momentum определены три основные роли пользователей, каждая из которых имеет специфический набор доступных операций.

Роль гостя представляет неаутентифицированного пользователя, который имеет ограниченный доступ к функционалу системы. Гость может просматривать главную страницу приложения с описанием возможностей и преимуществ платформы. Доступна регистрация нового аккаунта с указанием email, пароля и уникального никнейма. Гость может выполнить вход в систему, используя существующие учетные данные. После успешной аутентификации гость получает роль участника и доступ к полному функционалу приложения. Гость также может просматривать публичные профили зарегистрированных пользователей и открытую информацию о публичных командах.

Роль участника представляет аутентифицированного пользователя с базовыми правами в системе. Участник может управлять своим профилем, включая редактирование персональной информации, загрузку аватара, изменение биографии и настройку фонового изображения профиля. Доступно изменение пароля с подтверждением текущего пароля для обеспечения безопасности. Участник может просматривать и настраивать параметры приложения, включая выбор темы оформления, настройку фонового изображения и расположение виджетов на главной странице.

В контексте работы с командами участник может создавать новые команды с настройкой параметров приватности и требований к присоединению. Доступен просмотр списка всех публичных команд и поиск команд по названию или описанию. Участник может присоединяться к публичным командам без дополнительных подтверждений или к закрытым командам по коду приглашения. Возможен выход из команды, в которой участник не является владельцем. Участник может просматривать детальную информацию о командах, включая список участников, активные челленджи и статистику команды.

В рамках участия в челленджах участник может просматривать все челленджи своих команд с фильтрацией по статусу. Доступен просмотр детальной информации о челлендже, включая описание, даты проведения, список привычек и таблицу участников. Участник может отслеживать выполнение своих привычек, отмечая их статус на ежедневной основе. Поддерживается добавление заметок к каждой отметке выполнения для фиксации дополнительной информации. Участник может просматривать свою статистику по челленджу, включая процент выполнения, текущую серию и позицию в лидерборде.

Социальные функции для участника включают возможность создания постов в командах с добавлением текста и медиафайлов. Участник может редактировать и удалять свои посты. Доступно участие в чате команды с отправкой сообщений и просмотром истории переписки. Участник может просматривать профили других участников команды и их статистику. Система уведомлений информирует участника о важных событиях, таких как начало новых челленджей, достижения в команде и упоминания в постах или сообщениях.

Роль администратора команды расширяет права участника дополнительными возможностями управления контентом команды. Администратор может создавать новые челленджи с указанием названия, описания, дат проведения и загрузкой постера. Доступно редактирование существующих челленджей, включая изменение всех параметров и добавление новых привычек. Администратор может удалять челленджи, что также приводит к удалению всех связанных привычек и логов выполнения.

В контексте управления привычками администратор может создавать новые привычки в рамках челленджей с указанием названия и описания. Доступно редактирование существующих привычек и их удаление при необходимости. Администратор имеет право отмечать пропуски с разрешением для участников команды в случае обоснованных причин невыполнения привычки. Это позволяет учитывать форс-мажорные обстоятельства без негативного влияния на статистику участника.

Администратор может модерировать контент команды, включая удаление постов и сообщений, нарушающих правила команды. Доступен просмотр детальной статистики по всем участникам команды и их активности в челленджах. Администратор может редактировать настройки команды, включая изменение названия, описания и фонового изображения. Однако изменение параметров приватности и требований к присоединению доступно только владельцу команды.

Роль владельца команды включает все права администратора и дополнительные возможности управления командой. Владелец может назначать и снимать администраторов из числа участников команды. Доступно изменение всех параметров команды, включая настройки приватности, требования к коду приглашения и whitelist email. Владелец может создавать коды приглашений с настройкой максимального количества использований и срока действия. Доступно управление whitelist email, включая добавление и удаление адресов электронной почты.

Владелец команды может удалять участников из команды, что приводит к потере их доступа ко всем челленджам и контенту команды. Доступна передача владения командой другому участнику, после чего бывший владелец получает роль администратора. Владелец может удалить команду полностью, что приводит к удалению всех связанных данных, включая челленджи, привычки, логи выполнения, посты и сообщения. Перед удалением система запрашивает подтверждение для предотвращения случайного удаления.

### 2.2.2 Диаграмма компонентов системы

Диаграмма компонентов описывает физическую структуру системы и взаимосвязи между основными программными компонентами. Архитектура приложения Momentum состоит из нескольких уровней, каждый из которых выполняет специфические функции и взаимодействует с другими уровнями через определенные интерфейсы.

Уровень представления включает клиентское веб-приложение, разработанное на базе Next.js и React. Этот компонент отвечает за отображение пользовательского интерфейса, обработку пользовательского ввода и взаимодействие с серверным API. Клиентское приложение организовано в виде набора страниц и компонентов, каждый из которых представляет определенную функциональность. Страницы включают главную страницу, страницы аутентификации, страницу профиля, страницы команд, страницы челленджей и страницы настроек. Компоненты разделены на несколько категорий, включая базовые UI компоненты из библиотеки shadcn/ui, компоненты макета, компоненты для работы с командами, компоненты для работы с челленджами и специализированные компоненты визуализации.

Управление состоянием клиентского приложения осуществляется через несколько Zustand stores. AuthStore управляет состоянием аутентификации, хранит информацию о текущем пользователе и токены доступа. ThemeStore управляет настройками темы оформления и фонового изображения. UIStore хранит состояние пользовательского интерфейса, включая конфигурацию виджетов и временные данные. Все stores поддерживают персистентность данных через localStorage, что обеспечивает сохранение состояния между сессиями.

API клиент представляет собой набор модулей для взаимодействия с серверным API. Базовый модуль client предоставляет функцию apiRequest для выполнения HTTP-запросов с автоматической обработкой ошибок и обновлением токенов. Специализированные модули включают authApi для операций аутентификации, teamsApi для работы с командами, challengesApi для работы с челленджами, habitsApi для работы с привычками, habitLogsApi для работы с логами выполнения, postsApi для работы с постами, chatApi для работы с чатами, usersApi для работы с пользователями и storageApi для работы с файлами.

Уровень приложения представлен серверным приложением на базе NestJS, которое реализует бизнес-логику системы. Серверное приложение организовано в виде модулей, каждый из которых инкапсулирует связанную функциональность. AuthModule обеспечивает аутентификацию и авторизацию пользователей, включая регистрацию, вход, обновление токенов и управление сессиями. UsersModule управляет профилями пользователей, включая редактирование информации, изменение пароля и получение статистики.

TeamsModule обеспечивает управление командами, включая создание, редактирование, удаление команд, управление участниками и настройку параметров доступа. TeamMembersService управляет членством в командах и ролями участников. TeamInvitesService обеспечивает создание и валидацию кодов приглашений. TeamWhitelistService управляет whitelist email для закрытых команд. ChallengesModule управляет челленджами, включая создание, редактирование, удаление и получение статистики. HabitsModule управляет привычками в рамках челленджей. HabitLogsModule обеспечивает трекинг выполнения привычек, включая создание, обновление и удаление логов.

PostsModule управляет постами в командах, включая создание, редактирование и удаление. ChatsModule обеспечивает функционал чата команды с поддержкой пагинации сообщений. StorageModule управляет хранением файлов через интеграцию с MinIO. UserSettingsModule управляет пользовательскими настройками, включая тему оформления, расположение виджетов и другие параметры персонализации.

Каждый модуль состоит из нескольких компонентов. Controller обрабатывает HTTP-запросы, валидирует входные данные через DTO и вызывает соответствующие методы сервиса. Service содержит бизнес-логику и взаимодействует с базой данных через Prisma. DTO определяют структуру входных данных и правила валидации. Guards обеспечивают проверку прав доступа перед выполнением операций.

Уровень данных включает несколько компонентов для хранения различных типов информации. PostgreSQL используется в качестве основной реляционной базы данных для хранения структурированных данных. Prisma ORM обеспечивает типобезопасный доступ к базе данных и управление схемой через миграции. MinIO используется для хранения файлов, включая аватары пользователей, обложки команд, постеры челленджей и медиафайлы в постах и сообщениях.

Взаимодействие между компонентами осуществляется через четко определенные интерфейсы. Клиентское приложение взаимодействует с серверным приложением через RESTful API по протоколу HTTPS. Все запросы к защищенным эндпоинтам включают JWT токен в заголовке Authorization. Серверное приложение взаимодействует с базой данных через Prisma Client, который генерирует оптимизированные SQL-запросы. Для работы с файлами серверное приложение использует S3-совместимый API MinIO.

### 2.2.3 Диаграмма последовательности

Диаграмма последовательности описывает взаимодействие компонентов системы во времени для выполнения конкретных операций. Рассмотрим несколько ключевых сценариев работы системы, которые демонстрируют типичные паттерны взаимодействия.

Сценарий аутентификации пользователя начинается с того, что пользователь вводит свои учетные данные на странице входа и отправляет форму. Клиентское приложение выполняет локальную валидацию введенных данных, проверяя формат email и наличие пароля. После успешной валидации клиент отправляет POST-запрос на эндпоинт /api/auth/login с email и паролем в теле запроса. Запрос поступает в AuthController, который передает данные в AuthService для обработки.

AuthService выполняет поиск пользователя в базе данных по указанному email через Prisma Client. Если пользователь не найден, сервис возвращает ошибку с кодом 401 Unauthorized. Если пользователь найден, сервис извлекает хеш пароля из базы данных и сравнивает его с введенным паролем, используя библиотеку Argon2. При несовпадении паролей возвращается ошибка 401 Unauthorized. При успешной проверке пароля AuthService генерирует пару токенов: access token и refresh token.

Для генерации токенов сервис сначала создает запись сессии в базе данных через SessionsService, сохраняя информацию о пользователе, User-Agent браузера и IP-адресе. После создания сессии генерируется access token с коротким сроком действия, содержащий идентификатор пользователя, email и никнейм. Затем генерируется refresh token с длительным сроком действия, содержащий идентификатор пользователя и идентификатор сессии. Refresh token сохраняется в записи сессии в базе данных для последующей валидации.

AuthService возвращает AuthController объект с данными пользователя и обоими токенами. Controller формирует HTTP-ответ с кодом 200 OK и отправляет его клиенту. Клиентское приложение получает ответ и сохраняет данные пользователя и токены в AuthStore. Токены также сохраняются в localStorage для персистентности между сессиями. После успешного сохранения клиент перенаправляет пользователя на страницу профиля.

Сценарий создания челленджа демонстрирует более сложное взаимодействие с проверкой прав доступа. Пользователь, являющийся администратором или владельцем команды, заполняет форму создания челленджа с указанием названия, описания, дат начала и окончания. После заполнения формы клиент выполняет валидацию данных, проверяя, что дата окончания позже даты начала и что все обязательные поля заполнены. Клиент отправляет POST-запрос на эндпоинт /api/challenges с данными челленджа и access token в заголовке Authorization.

Запрос поступает в ChallengesController, где сначала срабатывает JwtAuthGuard, который извлекает токен из заголовка и валидирует его через JwtStrategy. Стратегия декодирует токен, извлекает идентификатор пользователя и запрашивает данные пользователя из базы через AuthService. Если токен валиден и пользователь существует, Guard прикрепляет объект пользователя к запросу и пропускает его дальше. Если токен невалиден или истек, Guard возвращает ошибку 401 Unauthorized.

После успешной аутентификации срабатывает TeamRolesGuard, который проверяет права пользователя в команде. Guard извлекает идентификатор команды из параметров запроса и запрашивает информацию о членстве пользователя в команде через TeamMembersService. Если пользователь не является членом команды или его роль не соответствует требуемой, Guard возвращает ошибку 403 Forbidden. При наличии необходимых прав Guard прикрепляет информацию о роли к запросу и пропускает его дальше.

Controller передает валидированные данные в ChallengesService для создания челленджа. Сервис выполняет дополнительную валидацию бизнес-правил, проверяя, что даты находятся в будущем и что команда существует. Сервис создает запись челленджа в базе данных через Prisma Client, указывая идентификатор команды, название, описание, даты и URL постера, если он был загружен. После успешного создания Prisma возвращает объект созданного челленджа с автоматически сгенерированным идентификатором и временными метками.

ChallengesService возвращает созданный челлендж Controller, который формирует HTTP-ответ с кодом 201 Created и отправляет его клиенту. Клиентское приложение получает ответ, отображает уведомление об успешном создании челленджа и обновляет список челленджей команды. Пользователь автоматически перенаправляется на страницу созданного челленджа, где может начать добавлять привычки.

Сценарий отслеживания выполнения привычки демонстрирует работу с логами. Пользователь открывает страницу челленджа и видит таблицу отслеживания привычек. Для отметки выполнения привычки пользователь кликает на соответствующую ячейку таблицы и выбирает статус из контекстного меню. Клиент отправляет POST-запрос на эндпоинт /api/habit-logs с данными о привычке, дате и выбранном статусе.

После прохождения аутентификации запрос поступает в HabitLogsController, который передает данные в HabitLogsService. Сервис выполняет валидацию, проверяя, что привычка существует, дата находится в пределах периода челленджа и пользователь является членом команды. Для обычных участников проверяется, что они отмечают только свои собственные привычки и не используют статус SKIPPED_EXCUSED. Для администраторов разрешается отмечать пропуски с разрешением для любых участников.

После валидации сервис выполняет операцию upsert через Prisma, которая создает новый лог, если запись не существует, или обновляет существующий лог, если запись уже есть. Это обеспечивает идемпотентность операции и позволяет пользователю изменять статус выполнения. Prisma возвращает созданный или обновленный лог, который сервис передает обратно контроллеру. Контроллер отправляет ответ клиенту, который обновляет отображение ячейки таблицы, применяя соответствующую цветовую кодировку.

## 2.3 Диаграмма базы данных и взаимосвязь компонентов

Схема базы данных приложения Momentum спроектирована в соответствии с принципами реляционной модели данных и нормализована до третьей нормальной формы для минимизации избыточности и обеспечения целостности информации. База данных содержит четырнадцать основных таблиц, связанных между собой отношениями один-к-одному, один-ко-многим и многие-ко-многим.

Центральной сущностью схемы является таблица users, которая хранит информацию о зарегистрированных пользователях системы. Таблица содержит первичный ключ id типа UUID, обеспечивающий уникальную идентификацию каждого пользователя. Поля email и nickname имеют ограничение уникальности, что предотвращает создание дубликатов. Поле password_hash хранит хешированный пароль, вычисленный с использованием алгоритма Argon2. Опциональные поля avatar_url, bio и header_bg_url позволяют пользователям персонализировать свой профиль. Поле nickname_changed_at отслеживает дату последнего изменения никнейма для реализации ограничения частоты изменений. Временные метки created_at и updated_at автоматически управляются базой данных.

Таблица sessions хранит информацию об активных сессиях пользователей для реализации механизма аутентификации на основе JWT токенов. Каждая сессия связана с конкретным пользователем через внешний ключ user_id, ссылающийся на таблицу users. Поле refresh_token хранит refresh token для данной сессии, что позволяет валидировать запросы на обновление access token. Опциональные поля user_agent и ip_address сохраняют информацию о браузере и IP-адресе для идентификации устройства. Поле expires_at определяет время истечения сессии, после которого refresh token становится невалидным. Для оптимизации производительности созданы индексы по полям user_id и refresh_token.

Таблица teams представляет команды пользователей, которые являются основной единицей совместной работы в системе. Каждая команда имеет владельца, идентификатор которого хранится в поле owner_id с внешним ключом на таблицу users. Поля name, slogan и description содержат информацию о команде. Поле header_bg_url хранит URL фонового изображения команды. Булевы поля is_private, require_invite_code и require_email_whitelist определяют параметры доступа к команде. При удалении владельца команды срабатывает каскадное удаление, что обеспечивает целостность данных.

Таблица team_members реализует связь многие-ко-многим между пользователями и командами с дополнительными атрибутами. Каждая запись связывает пользователя с командой через внешние ключи team_id и user_id. Поле role типа enum определяет роль пользователя в команде и может принимать значения MEMBER, ADMIN или OWNER. Поле joined_at фиксирует дату вступления в команду. Уникальное ограничение на пару team_id и user_id предотвращает дублирование членства. Индексы по обоим внешним ключам обеспечивают быструю выборку участников команды и команд пользователя.

Таблица team_invites хранит коды приглашений для присоединения к командам. Каждый код связан с командой через внешний ключ team_id. Поле code содержит уникальный шестисимвольный код, генерируемый при создании приглашения. Опциональное поле max_uses ограничивает максимальное количество использований кода. Поле used_count отслеживает фактическое количество использований. Опциональное поле expires_at определяет дату истечения кода. Булево поле is_active позволяет деактивировать код без его удаления. Поле created_by хранит идентификатор пользователя, создавшего код. Индексы по полям team_id и code обеспечивают быструю валидацию кодов.

Таблица team_whitelist_emails хранит список разрешенных email-адресов для закрытых команд. Каждая запись связывает email с командой через внешний ключ team_id. Уникальное ограничение на пару team_id и email предотвращает дублирование адресов. Индексы по обоим полям обеспечивают быструю проверку наличия email в whitelist при попытке присоединения к команде.

Таблица challenges представляет испытания, организуемые в рамках команд. Каждый челлендж связан с командой через внешний ключ team_id. Поля title и description содержат информацию о челлендже. Поле poster_url хранит URL постера челленджа для визуального представления. Поля start_date и end_date типа DATE определяют период проведения челленджа. Индекс по полю team_id обеспечивает быструю выборку челленджей команды. Составной индекс по полям start_date и end_date оптимизирует фильтрацию челленджей по статусу.

Таблица habits хранит привычки, которые необходимо отслеживать в рамках челленджей. Каждая привычка связана с челленджем через внешний ключ challenge_id. Поля title и description определяют название и описание привычки. При удалении челленджа все связанные привычки удаляются каскадно, что обеспечивает целостность данных. Индекс по полю challenge_id оптимизирует выборку привычек челленджа.

Таблица habit_logs является центральной для функционала трекинга привычек и хранит записи о выполнении привычек пользователями. Каждый лог связан с привычкой через внешний ключ habit_id и с пользователем через внешний ключ user_id. Поле date типа DATE определяет дату выполнения. Поле status типа enum может принимать значения COMPLETED для полного выполнения, PARTIAL для частичного выполнения и SKIPPED_EXCUSED для пропуска с разрешением администратора. Опциональное поле note позволяет пользователю добавить заметку к записи. Уникальное ограничение на триаду habit_id, user_id и date обеспечивает, что для каждой комбинации привычки, пользователя и даты существует не более одной записи. Индексы по всем трем полям оптимизируют различные типы запросов к таблице.

Таблица chats представляет чаты команд для обмена сообщениями между участниками. Каждый чат связан с командой отношением один-к-одному через уникальный внешний ключ team_id. Это обеспечивает, что у каждой команды может быть только один чат. Таблица messages хранит сообщения в чатах. Каждое сообщение связано с чатом через внешний ключ chat_id и с автором через внешний ключ user_id. Поле content типа TEXT хранит текст сообщения. Опциональное поле media_url позволяет прикреплять медиафайлы к сообщениям. Индексы по полям chat_id, user_id и created_at оптимизируют выборку сообщений чата с сортировкой по времени.

Таблица posts хранит посты, публикуемые участниками в командах. Каждый пост связан с командой через внешний ключ team_id и с автором через внешний ключ author_id. Поля title и content содержат заголовок и текст поста. Поле media_urls типа JSONB хранит массив URL медиафайлов, прикрепленных к посту. Использование JSONB обеспечивает эффективное хранение и индексирование массивов переменной длины. Индексы по полям team_id, author_id и created_at оптимизируют выборку постов команды и постов автора с сортировкой по времени.

Таблица user_dashboards хранит конфигурацию дашборда пользователя. Связь с пользователем реализована через уникальный внешний ключ user_id, обеспечивающий отношение один-к-одному. Поле layout_config типа JSONB хранит конфигурацию расположения виджетов на главной странице пользователя. Использование JSONB позволяет хранить сложные структуры данных без необходимости создания дополнительных таблиц.

Таблица user_settings хранит пользовательские настройки приложения. Связь с пользователем также реализована через уникальный внешний ключ user_id. Поле settings типа JSONB хранит все настройки пользователя в виде JSON-объекта, включая тему оформления, язык интерфейса, настройки фона и расположение виджетов. Такой подход обеспечивает гибкость в добавлении новых настроек без изменения схемы базы данных.

Связи между таблицами реализованы через внешние ключи с определенными правилами каскадного удаления. При удалении пользователя каскадно удаляются все его сессии, членства в командах, логи выполнения привычек, сообщения, посты, дашборд и настройки. При удалении команды каскадно удаляются все членства, челленджи, чат, посты, коды приглашений и записи whitelist. При удалении челленджа каскадно удаляются все привычки и их логи выполнения. Такая структура каскадных удалений обеспечивает целостность данных и предотвращает появление осиротевших записей.

## 2.4 Описание информационных объектов и ограничений целостности

### 2.4.1 Сущность User

Сущность User представляет зарегистрированного пользователя системы и является центральной в архитектуре приложения. Первичный ключ id имеет тип UUID и генерируется автоматически при создании записи, что обеспечивает глобальную уникальность идентификаторов и предотвращает коллизии при возможном объединении баз данных. Использование UUID вместо автоинкрементных целых чисел также повышает безопасность, так как идентификаторы не являются последовательными и их сложнее предсказать.

Поле email типа VARCHAR с ограничением уникальности хранит адрес электронной почты пользователя. Перед сохранением email приводится к нижнему регистру для обеспечения регистронезависимого сравнения. На уровне приложения выполняется валидация формата email с использованием регулярных выражений. Поле password_hash типа VARCHAR хранит хеш пароля, вычисленный с использованием алгоритма Argon2 с параметрами, рекомендованными OWASP для защиты от атак перебора. Исходный пароль никогда не сохраняется в базе данных, что обеспечивает безопасность учетных данных даже в случае компрометации базы.

Поле nickname типа VARCHAR с ограничением уникальности хранит отображаемое имя пользователя. Никнейм должен содержать от трех до тридцати символов и может включать буквы латинского алфавита, цифры, дефисы и подчеркивания. Валидация никнейма выполняется на уровне приложения с использованием регулярного выражения. Поле nickname_changed_at типа TIMESTAMP отслеживает дату последнего изменения никнейма. Бизнес-правило ограничивает частоту изменения никнейма до одного раза в тридцать дней, что предотвращает злоупотребления и помогает другим пользователям идентифицировать участников.

Опциональное поле avatar_url типа VARCHAR хранит URL изображения аватара пользователя. URL указывает на файл в объектном хранилище MinIO. При удалении или изменении аватара старый файл удаляется из хранилища для освобождения пространства. Опциональное поле bio типа TEXT позволяет пользователю добавить краткую биографию или описание. Длина биографии ограничена пятьюстами символами на уровне приложения. Опциональное поле header_bg_url типа VARCHAR хранит URL фонового изображения профиля пользователя.

Поля created_at и updated_at типа TIMESTAMP автоматически управляются базой данных. Поле created_at устанавливается при создании записи и не изменяется в дальнейшем. Поле updated_at автоматически обновляется при любом изменении записи, что позволяет отслеживать время последней модификации.

Ограничения целостности для сущности User включают уникальность email и nickname, что обеспечивается уникальными индексами на уровне базы данных. Ограничение NOT NULL применяется к обязательным полям id, email, password_hash, nickname, created_at и updated_at. Каскадное удаление применяется ко всем связанным сущностям: при удалении пользователя автоматически удаляются все его сессии, членства в командах, логи выполнения привычек, сообщения, посты, дашборд и настройки.

### 2.4.2 Сущность Team

Сущность Team представляет команду пользователей для совместного участия в челленджах. Первичный ключ id имеет тип UUID и генерируется автоматически. Поле name типа VARCHAR хранит название команды длиной от трех до пятидесяти символов. Название не обязано быть уникальным глобально, что позволяет разным пользователям создавать команды с одинаковыми названиями. Опциональное поле slogan типа VARCHAR хранит краткий слоган команды длиной до ста символов. Опциональное поле description типа TEXT позволяет добавить подробное описание команды длиной до пятисот символов.

Поле owner_id типа UUID является внешним ключом, ссылающимся на таблицу users. Владелец команды имеет полный контроль над командой и может выполнять все операции управления. При удалении владельца команды срабатывает каскадное удаление самой команды, так как команда не может существовать без владельца. Опциональное поле header_bg_url типа VARCHAR хранит URL фонового изображения команды для визуального оформления страницы команды.

Булево поле is_private определяет, является ли команда приватной. Приватные команды не отображаются в общем списке команд и доступны только участникам. Публичные команды видны всем пользователям и отображаются в разделе поиска команд. Булево поле require_invite_code определяет, требуется ли код приглашения для присоединения к команде. Если флаг установлен, пользователи могут присоединиться к команде только по валидному коду приглашения. Булево поле require_email_whitelist определяет, требуется ли наличие email пользователя в whitelist для присоединения. Эти три флага могут комбинироваться для создания различных уровней доступа к команде.

Ограничения целостности включают обязательность полей id, name, owner_id, is_private, require_invite_code, require_email_whitelist, created_at и updated_at. Внешний ключ owner_id должен ссылаться на существующую запись в таблице users. Индекс по полю owner_id оптимизирует выборку команд конкретного владельца. Каскадное удаление применяется ко всем связанным сущностям: при удалении команды автоматически удаляются все членства, челленджи, чат, посты, коды приглашений и записи whitelist.

### 2.4.3 Сущность Challenge

Сущность Challenge представляет испытание, организуемое в рамках команды. Первичный ключ id имеет тип UUID. Поле team_id типа UUID является внешним ключом, ссылающимся на таблицу teams. Каждый челлендж принадлежит ровно одной команде, и участвовать в нем могут только члены этой команды. При удалении команды все ее челленджи удаляются каскадно.

Поле title типа VARCHAR хранит название челленджа длиной от трех до ста символов. Опциональное поле description типа TEXT позволяет добавить подробное описание целей и правил челленджа. Опциональное поле poster_url типа VARCHAR хранит URL постера челленджа для визуального представления на странице команды и в галерее челленджей.

Поля start_date и end_date типа DATE определяют период проведения челленджа. Использование типа DATE без времени упрощает логику работы с датами и обеспечивает консистентность при работе в различных часовых поясах. Бизнес-правило требует, чтобы дата окончания была строго позже даты начала. Валидация этого правила выполняется на уровне приложения перед сохранением записи. Статус челленджа определяется динамически на основе текущей даты: челлендж считается активным, если текущая дата находится между датами начала и окончания включительно, предстоящим, если текущая дата меньше даты начала, и завершенным, если текущая дата больше даты окончания.

Ограничения целостности включают обязательность всех полей кроме description и poster_url. Внешний ключ team_id должен ссылаться на существующую команду. Индекс по полю team_id оптимизирует выборку челленджей команды. Составной индекс по полям start_date и end_date оптимизирует фильтрацию челленджей по статусу. Каскадное удаление применяется ко всем привычкам челленджа и их логам выполнения.

### 2.4.4 Сущность Habit

Сущность Habit представляет привычку, которую необходимо отслеживать в рамках челленджа. Первичный ключ id имеет тип UUID. Поле challenge_id типа UUID является внешним ключом, ссылающимся на таблицу challenges. Каждая привычка принадлежит ровно одному челленджу. При удалении челленджа все его привычки удаляются каскадно вместе со всеми логами выполнения.

Поле title типа VARCHAR хранит название привычки длиной от трех до ста символов. Название должно быть достаточно описательным, чтобы участники понимали, что именно нужно выполнять. Опциональное поле description типа TEXT позволяет добавить подробное описание привычки, включая критерии выполнения, рекомендации и дополнительную информацию.

Ограничения целостности включают обязательность полей id, challenge_id, title, created_at и updated_at. Внешний ключ challenge_id должен ссылаться на существующий челлендж. Индекс по полю challenge_id оптимизирует выборку привычек челленджа. Каскадное удаление применяется ко всем логам выполнения привычки.

### 2.4.5 Сущность HabitLog

Сущность HabitLog представляет запись о выполнении привычки конкретным пользователем в конкретную дату. Эта сущность является центральной для функционала трекинга привычек. Первичный ключ id имеет тип UUID. Поле habit_id типа UUID является внешним ключом, ссылающимся на таблицу habits. Поле user_id типа UUID является внешним ключом, ссылающимся на таблицу users. Поле date типа DATE определяет дату выполнения привычки.

Поле status типа ENUM может принимать одно из трех значений. Значение COMPLETED указывает на полное выполнение привычки в указанную дату. Значение PARTIAL указывает на частичное выполнение привычки, когда пользователь выполнил часть требуемых действий, но не полностью. Значение SKIPPED_EXCUSED указывает на пропуск выполнения с разрешением администратора, что используется в случае обоснованных причин невыполнения, таких как болезнь или форс-мажорные обстоятельства. Обычные участники не могут устанавливать статус SKIPPED_EXCUSED для своих привычек, это право доступно только администраторам и владельцам команды.

Опциональное поле note типа TEXT позволяет пользователю добавить заметку к записи выполнения. Заметка может содержать дополнительную информацию о выполнении, возникших сложностях или достигнутых результатах. Длина заметки ограничена пятьюстами символами на уровне приложения.

Ключевым ограничением целостности для сущности HabitLog является уникальное ограничение на триаду полей habit_id, user_id и date. Это ограничение обеспечивает, что для каждой комбинации привычки, пользователя и даты может существовать не более одной записи. При попытке создания дубликата выполняется операция обновления существующей записи вместо создания новой. Такой подход обеспечивает идемпотентность операций трекинга и позволяет пользователям изменять статус выполнения.

Дополнительные ограничения включают обязательность всех полей кроме note. Внешние ключи habit_id и user_id должны ссылаться на существующие записи. Индексы по полям habit_id, user_id и date оптимизируют различные типы запросов: выборку всех логов привычки, выборку всех логов пользователя и выборку логов за определенную дату. Каскадное удаление применяется при удалении привычки или пользователя.

### 2.4.6 Вспомогательные сущности

Сущность Session хранит информацию об активных сессиях пользователей. Ключевым полем является refresh_token, который используется для валидации запросов на обновление access token. Поле expires_at определяет время истечения сессии, после которого refresh token становится невалидным. Периодическая задача очистки удаляет истекшие сессии из базы данных для освобождения пространства.

Сущность TeamMember реализует связь многие-ко-многим между пользователями и командами. Поле role типа ENUM определяет роль пользователя в команде и может принимать значения MEMBER, ADMIN или OWNER. Уникальное ограничение на пару team_id и user_id предотвращает дублирование членства. Бизнес-правило требует, чтобы в каждой команде был ровно один владелец, что обеспечивается на уровне приложения.

Сущность TeamInvite хранит коды приглашений для присоединения к командам. Поле code содержит уникальный шестисимвольный код, генерируемый из алфавита, исключающего визуально похожие символы. Поле max_uses ограничивает количество использований кода, а поле used_count отслеживает фактическое использование. Бизнес-правило запрещает использование кода, если used_count достиг max_uses или текущая дата превысила expires_at.

Сущность TeamWhitelistEmail хранит разрешенные email-адреса для закрытых команд. Уникальное ограничение на пару team_id и email предотвращает дублирование. При попытке присоединения к команде с требованием whitelist система проверяет наличие email пользователя в этой таблице.

Сущности Chat и Message реализуют функционал чата команды. Отношение один-к-одному между Chat и Team обеспечивается уникальным внешним ключом. Сообщения связаны с чатом и автором через соответствующие внешние ключи. Индекс по полю created_at оптимизирует выборку сообщений в хронологическом порядке.

Сущность Post представляет посты в командах. Поле media_urls типа JSONB хранит массив URL медиафайлов переменной длины. Использование JSONB обеспечивает эффективное хранение и возможность индексирования содержимого массива при необходимости.

Сущности UserDashboard и UserSettings хранят персональные настройки пользователей. Использование типа JSONB для полей layout_config и settings обеспечивает гибкость в структуре данных и позволяет добавлять новые настройки без изменения схемы базы данных.

## 2.5 Проектирование основных алгоритмов

### 2.5.1 Алгоритм аутентификации и обновления токенов

Алгоритм аутентификации реализован на основе JWT токенов с механизмом автоматического обновления для обеспечения непрерывной работы пользователя. Система использует два типа токенов: access token с коротким сроком действия для доступа к API и refresh token с длительным сроком действия для обновления access token.

Процесс регистрации начинается с получения данных пользователя, включая email, пароль и никнейм. Система выполняет валидацию формата email с использованием регулярного выражения, проверяет сложность пароля и уникальность никнейма. Пароль хешируется с использованием алгоритма Argon2 с параметрами, включающими фактор памяти, фактор времени и степень параллелизма, рекомендованными OWASP. Создается запись пользователя в базе данных с хешированным паролем. После успешного создания пользователя система генерирует пару токенов и создает первую сессию, после чего возвращает данные пользователя и токены клиенту.

Процесс входа начинается с получения email и пароля от пользователя. Система выполняет поиск пользователя по email в базе данных. Если пользователь не найден, возвращается ошибка с общим сообщением о неверных учетных данных для предотвращения перечисления пользователей. Если пользователь найден, система извлекает хеш пароля и сравнивает его с введенным паролем, используя функцию верификации Argon2. При несовпадении паролей возвращается та же ошибка о неверных учетных данных. При успешной проверке пароля система генерирует новую пару токенов и создает новую сессию, после чего возвращает данные пользователя и токены клиенту.

Генерация токенов выполняется в несколько этапов. Сначала создается запись сессии в базе данных с временным refresh token для получения идентификатора сессии. Access token генерируется с payload, содержащим идентификатор пользователя, email и никнейм. Срок действия access token устанавливается в пятнадцать минут. Токен подписывается секретным ключом с использованием алгоритма HS256. Refresh token генерируется с payload, содержащим идентификатор пользователя и идентификатор сессии. Срок действия refresh token устанавливается в семь дней. Токен подписывается отдельным секретным ключом. Сгенерированный refresh token сохраняется в записи сессии в базе данных для последующей валидации. Оба токена возвращаются клиенту.

Процесс обновления токенов начинается с получения refresh token от клиента. Система декодирует и верифицирует refresh token с использованием секретного ключа для refresh токенов. Если токен невалиден или истек, возвращается ошибка аутентификации. Из payload токена извлекается идентификатор сессии. Система выполняет поиск сессии в базе данных по идентификатору. Если сессия не найдена или истекла, возвращается ошибка аутентификации. Система сравнивает refresh token из запроса с токеном, сохраненным в сессии. Если токены не совпадают, это может указывать на компрометацию токена, и возвращается ошибка аутентификации.

При успешной валидации система удаляет старую сессию из базы данных, реализуя ротацию токенов. Создается новая сессия с новой парой токенов. Новый access token и refresh token возвращаются клиенту. Клиент сохраняет новые токены и использует их для последующих запросов. Такой подход обеспечивает безопасность за счет ограничения времени жизни токенов и предотвращения повторного использования refresh токенов.

Автоматическое обновление токенов реализовано на клиенте через перехватчик запросов. При получении ответа с кодом 401 Unauthorized клиент автоматически пытается обновить токен, используя сохраненный refresh token. Если обновление успешно, исходный запрос повторяется с новым access token. Если обновление не удалось, пользователь выходит из системы и перенаправляется на страницу входа. Такой механизм обеспечивает прозрачное обновление токенов без вмешательства пользователя.

### 2.5.2 Алгоритм отслеживания выполнения привычек

Алгоритм отслеживания выполнения привычек обеспечивает создание, обновление и валидацию логов выполнения с учетом прав доступа пользователей. Процесс начинается с получения данных о выполнении привычки от клиента, включая идентификатор привычки, дату выполнения, статус и опциональную заметку. Для администраторов дополнительно может быть указан идентификатор целевого пользователя для логирования выполнения за другого участника.

Первым этапом валидации является проверка существования привычки в базе данных. Система выполняет запрос к таблице habits по указанному идентификатору с включением связанного челленджа и команды. Если привычка не найдена, возвращается ошибка с кодом 404 Not Found. Если привычка найдена, извлекается информация о челлендже, включая даты начала и окончания, и информация о команде.

Следующим этапом является валидация даты выполнения. Система проверяет, что указанная дата находится в пределах периода челленджа, то есть больше или равна дате начала и меньше или равна дате окончания. Если дата выходит за пределы периода, возвращается ошибка с сообщением о недопустимой дате. Эта проверка предотвращает создание логов для дат, когда челлендж еще не начался или уже завершился.

Третьим этапом является проверка прав доступа. Если в запросе указан идентификатор целевого пользователя, отличный от идентификатора текущего пользователя, система проверяет роль текущего пользователя в команде. Для этого выполняется запрос к таблице team_members по идентификаторам команды и пользователя. Если пользователь не является членом команды, возвращается ошибка с кодом 403 Forbidden. Если пользователь является обычным участником, проверяется статус выполнения. Обычные участники могут устанавливать только статусы COMPLETED и PARTIAL только для себя. Попытка установить статус SKIPPED_EXCUSED или логировать выполнение за другого пользователя приводит к ошибке с кодом 403 Forbidden.

Администраторы и владельцы команды имеют расширенные права. Они могут устанавливать статус SKIPPED_EXCUSED для любых участников команды. Это используется в случаях, когда участник не смог выполнить привычку по обоснованным причинам, таким как болезнь или форс-мажорные обстоятельства. Однако даже администраторы не могут устанавливать статусы COMPLETED или PARTIAL за других пользователей, так как это нарушает принцип честности трекинга.

После успешной валидации система выполняет операцию upsert в таблицу habit_logs. Операция upsert пытается создать новую запись с указанными данными. Если запись с такой комбинацией habit_id, user_id и date уже существует, выполняется обновление существующей записи. Это обеспечивается уникальным ограничением на триаду этих полей. Обновляются поля status, note и updated_at. Если запись не существует, создается новая запись со всеми указанными полями и автоматически генерируемым идентификатором.

После успешного выполнения операции система возвращает созданный или обновленный лог клиенту. Клиент обновляет отображение соответствующей ячейки в таблице трекинга, применяя цветовую кодировку в соответствии со статусом. Зеленый цвет используется для статуса COMPLETED, желтый для статуса PARTIAL и серый для статуса SKIPPED_EXCUSED. Пустые ячейки остаются без цвета, указывая на отсутствие записи выполнения.

Массовое обновление логов реализовано через отдельный эндпоинт, принимающий массив объектов с данными логов. Система последовательно обрабатывает каждый объект, выполняя те же этапы валидации и сохранения. Результаты обработки каждого лога возвращаются в виде массива, где каждый элемент содержит либо успешно созданный лог, либо информацию об ошибке. Такой подход позволяет клиенту эффективно обновлять множество логов одним запросом, например, при заполнении таблицы за несколько дней.

### 2.5.3 Алгоритм расчета статистики и лидерборда

Алгоритм расчета статистики челленджа выполняет агрегацию данных о выполнении привычек всеми участниками для предоставления детальной информации о прогрессе. Процесс начинается с получения идентификатора челленджа и загрузки полной информации о челлендже из базы данных, включая даты начала и окончания, список привычек и информацию о команде.

Первым этапом является расчет временных метрик. Система вычисляет общее количество дней челленджа как разницу между датами окончания и начала плюс один. Вычисляется количество прошедших дней как разницу между текущей датой и датой начала, ограниченную общим количеством дней. Если челлендж еще не начался, количество прошедших дней равно нулю. Если челлендж уже завершился, количество прошедших дней равно общему количеству дней.

Вторым этапом является подсчет участников челленджа. Система выполняет запрос к таблице team_members для получения всех членов команды, к которой принадлежит челлендж. Количество участников используется для расчета ожидаемого количества логов. Подсчитывается количество привычек в челлендже из связанной коллекции habits.

Третьим этапом является агрегация логов выполнения. Система выполняет запрос к таблице habit_logs для получения всех логов, связанных с привычками данного челленджа. Запрос включает группировку по статусу для подсчета количества логов каждого типа. Результат включает общее количество логов, количество логов со статусом COMPLETED, количество логов со статусом PARTIAL и количество логов со статусом SKIPPED_EXCUSED.

Четвертым этапом является расчет метрик выполнения. Ожидаемое количество логов вычисляется как произведение количества участников, количества привычек и количества прошедших дней. Процент выполнения вычисляется как отношение количества выполненных логов к ожидаемому количеству логов, умноженное на сто. Для более точной оценки можно использовать взвешенный процент, где полностью выполненные логи имеют вес единица, частично выполненные логи имеют вес ноль целых пять десятых, а пропущенные с разрешением имеют вес ноль.

Алгоритм формирования лидерборда выполняет ранжирование участников челленджа на основе их результатов. Процесс начинается с получения всех членов команды челленджа. Для каждого участника выполняется агрегация его логов выполнения привычек данного челленджа. Подсчитывается количество логов со статусом COMPLETED и количество логов со статусом PARTIAL. Вычисляется общий счет участника как сумма количества полностью выполненных логов и половины количества частично выполненных логов. Такая формула стимулирует полное выполнение привычек, но также учитывает частичное выполнение.

Дополнительно для каждого участника может вычисляться текущая серия последовательных дней выполнения. Алгоритм расчета серии начинается с текущей даты и движется назад по дням, проверяя наличие выполненных логов. Серия прерывается при обнаружении дня без выполненных логов или с пропущенными привычками. Дни со статусом SKIPPED_EXCUSED не прерывают серию, так как пропуск был разрешен администратором.

После вычисления метрик для всех участников система сортирует их по убыванию общего счета. При равенстве счетов применяется дополнительная сортировка по текущей серии, затем по дате вступления в команду. Каждому участнику присваивается ранг на основе его позиции в отсортированном списке. Участники с одинаковым счетом получают одинаковый ранг. Сформированный лидерборд возвращается клиенту для отображения в таблице и 3D визуализации горы.

## 2.6 Архитектура веб-сервиса

### 2.6.1 Архитектура серверной части

Серверная часть приложения построена на основе фреймворка NestJS, который реализует архитектурный паттерн Model-View-Controller с дополнительным слоем сервисов для инкапсуляции бизнес-логики. Архитектура организована в виде модулей, каждый из которых представляет собой независимую единицу функциональности с четко определенными границами и интерфейсами взаимодействия.

Точкой входа в приложение является файл main.ts, который выполняет инициализацию NestJS приложения и настройку глобальных параметров. Устанавливается глобальный префикс api для всех маршрутов, что обеспечивает логическое разделение API от других возможных эндпоинтов. Настраивается CORS для разрешения кросс-доменных запросов с клиентского приложения. Регистрируется глобальный ValidationPipe для автоматической валидации всех входящих данных на основе DTO. Настраивается Swagger для автоматической генерации документации API на основе декораторов в контроллерах и DTO.

Корневой модуль AppModule импортирует все функциональные модули и настраивает глобальные сервисы. ConfigModule обеспечивает доступ к переменным окружения через ConfigService. PrismaModule предоставляет PrismaService для взаимодействия с базой данных. Все функциональные модули регистрируются в массиве imports корневого модуля.

Каждый функциональный модуль следует единой структуре. Модуль декларирует свои контроллеры в массиве controllers и сервисы в массиве providers. Зависимости от других модулей указываются в массиве imports. Сервисы, которые должны быть доступны другим модулям, перечисляются в массиве exports. Такая структура обеспечивает инкапсуляцию и явное определение зависимостей.

Контроллеры отвечают за обработку HTTP-запросов и формирование ответов. Каждый контроллер аннотируется декоратором Controller с указанием базового пути для всех его маршрутов. Методы контроллера аннотируются декораторами HTTP-методов, такими как Get, Post, Put, Patch и Delete. Параметры методов извлекаются из запроса с помощью декораторов Param, Query, Body и CurrentUser. Контроллеры выполняют минимальную обработку, делегируя всю бизнес-логику сервисам.

Сервисы содержат бизнес-логику приложения и взаимодействуют с базой данных через Prisma Client. Каждый сервис аннотируется декоратором Injectable, что позволяет использовать его в системе внедрения зависимостей. Сервисы получают зависимости через конструктор, включая PrismaService для доступа к базе данных и другие сервисы для взаимодействия с другими модулями. Методы сервисов выполняют валидацию бизнес-правил, обращаются к базе данных и возвращают результаты или выбрасывают исключения при ошибках.

DTO определяют структуру входных данных и правила валидации. Каждый DTO является классом с полями, аннотированными декораторами валидации из библиотеки class-validator. Декораторы включают IsString, IsEmail, IsNotEmpty, MinLength, MaxLength, IsOptional, IsBoolean, IsEnum, IsDateString и другие. ValidationPipe автоматически валидирует входные данные на основе этих декораторов и возвращает детальные сообщения об ошибках при невалидных данных.

Guards обеспечивают проверку прав доступа перед выполнением операций. JwtAuthGuard проверяет наличие и валидность JWT токена в заголовке Authorization. Guard извлекает токен, декодирует его и валидирует подпись. При успешной валидации Guard прикрепляет данные пользователя к объекту запроса. TeamRolesGuard проверяет роль пользователя в команде на основе метаданных, установленных декоратором Roles. Guard извлекает идентификатор команды из параметров запроса, находит членство пользователя в команде и проверяет соответствие роли требуемой.

Декораторы обеспечивают декларативное управление поведением маршрутов. Декоратор Public помечает маршруты, не требующие аутентификации. Декоратор Roles указывает требуемые роли для доступа к маршруту. Декоратор CurrentUser извлекает данные текущего пользователя из объекта запроса. Использование декораторов делает код контроллеров более читаемым и декларативным.

Обработка ошибок реализована через систему исключений NestJS. Сервисы выбрасывают стандартные HTTP-исключения, такие как NotFoundException, ForbiddenException, ConflictException, BadRequestException и UnauthorizedException. Глобальный фильтр исключений перехватывает все исключения и формирует единообразные ответы с соответствующими HTTP-кодами и сообщениями об ошибках.

### 2.6.2 Архитектура клиентской части

Клиентская часть приложения построена на основе фреймворка Next.js с использованием App Router для организации маршрутизации. Архитектура следует компонентному подходу, где каждый компонент представляет собой независимую единицу пользовательского интерфейса с собственной логикой и состоянием.

Структура приложения организована в директории app, где каждая поддиректория представляет маршрут. Группы маршрутов, такие как auth и dashboard, организованы в директории с круглыми скобками, что исключает их из URL. Динамические маршруты используют квадратные скобки для параметров, например teamId и challengeId. Каждый маршрут содержит файл page.tsx, который экспортирует компонент страницы.

Компоненты организованы в директории components с разделением по функциональным областям. Директория ui содержит базовые компоненты из библиотеки shadcn/ui, такие как Button, Card, Dialog, Input и другие. Директория layout содержит компоненты макета, включая Header, Sidebar и Hero. Директории challenge, team, workspace и mountain содержат специализированные компоненты для соответствующих функциональных областей.

Управление состоянием реализовано через Zustand stores. AuthStore управляет состоянием аутентификации, хранит данные пользователя и токены. Store использует middleware persist для автоматического сохранения состояния в localStorage. При восстановлении состояния из localStorage устанавливается флаг isHydrated, что позволяет компонентам определить, когда состояние полностью загружено. ThemeStore управляет настройками темы и фона. UIStore хранит временное состояние пользовательского интерфейса.

API клиент организован в директории lib/api с разделением на модули по функциональным областям. Базовый модуль client предоставляет функцию apiRequest, которая является оберткой над fetch API. Функция автоматически добавляет access token в заголовок Authorization для защищенных запросов. При получении ответа с кодом 401 функция автоматически пытается обновить токен через функцию refreshAccessToken. Если обновление успешно, исходный запрос повторяется с новым токеном. Если обновление не удалось, пользователь выходит из системы.

Специализированные API модули используют базовую функцию apiRequest для выполнения запросов к конкретным эндпоинтам. Каждый модуль экспортирует объект с методами для различных операций. Методы принимают типизированные параметры и возвращают типизированные результаты, что обеспечивает типобезопасность на уровне компиляции.

Защита маршрутов реализована через компонент AuthProvider, который оборачивает все приложение. Провайдер проверяет состояние аутентификации при монтировании и перенаправляет неаутентифицированных пользователей на страницу входа. Публичные маршруты, такие как главная страница и страницы аутентификации, исключены из проверки. Layout компонент dashboard дополнительно проверяет аутентификацию и отображает индикатор загрузки до завершения проверки.

Формы реализованы с использованием библиотеки React Hook Form в связке с Zod для валидации. Схемы валидации определяются с использованием Zod и используются для автоматической валидации полей формы. Компоненты Form из shadcn/ui обеспечивают единообразное отображение полей формы с сообщениями об ошибках.

Стилизация выполнена с использованием Tailwind CSS с utility-first подходом. Компоненты используют классы Tailwind для определения стилей. Функция cn из библиотеки clsx объединяет классы и разрешает конфликты. Глобальные стили и CSS-переменные для темы определены в файле globals.css.

Оптимизация производительности достигается через несколько механизмов. Серверные компоненты используются для статического контента, что улучшает время первой загрузки. Клиентские компоненты используются для интерактивных элементов с пометкой use client. Ленивая загрузка применяется для тяжелых компонентов, таких как 3D визуализация. Мемоизация через useMemo и useCallback предотвращает ненужные перерисовки.

Спроектированные алгоритмы обеспечивают реализацию ключевых функций системы. Алгоритм аутентификации реализует безопасный механизм входа с использованием JWT токенов и автоматическим обновлением. Алгоритм отслеживания выполнения привычек обеспечивает валидацию данных и проверку прав доступа. Алгоритм расчета статистики выполняет агрегацию данных для предоставления детальной информации о прогрессе.

Описание архитектуры веб-сервиса детализирует структуру серверной и клиентской частей приложения. Серверная архитектура на базе NestJS обеспечивает модульность, тестируемость и масштабируемость. Клиентская архитектура на базе Next.js обеспечивает высокую производительность и удобство разработки. Разделение ответственности между компонентами обеспечивает поддерживаемость и возможность параллельной разработки.

Разработанная архитектура обеспечивает основу для реализации всех функциональных требований, определенных в первой главе. Модульная структура позволяет легко расширять функционал и модифицировать существующие компоненты.
