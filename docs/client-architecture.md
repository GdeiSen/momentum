## Обзор архитектуры

Клиентское приложение Momentum построено на фреймворке **Next.js 16** с использованием **React 19** и **TypeScript**. Приложение следует современным практикам разработки веб-приложений, используя серверные и клиентские компоненты, оптимизацию производительности и удобство использования.

### Основные принципы архитектуры

1. **Компонентный подход** - приложение разбито на переиспользуемые компоненты
2. **Разделение ответственности** - четкое разделение между страницами, компонентами, API клиентами и stores
3. **Типобезопасность** - полная типизация с TypeScript
4. **Управление состоянием** - централизованное управление через Zustand stores
5. **Оптимизация производительности** - использование React hooks, мемоизации и ленивой загрузки
6. **Адаптивность** - поддержка различных размеров экранов и устройств

## Структура приложения

### Организация файлов

Приложение использует структуру Next.js App Router, где маршруты определяются структурой папок в директории `app/`.

#### Основные директории:

- **`app/`** - страницы и маршруты приложения
  - **`(auth)/`** - страницы аутентификации (login, register)
  - **`(dashboard)/`** - защищенные страницы дашборда
- **`components/`** - переиспользуемые React компоненты
  - **`ui/`** - базовые UI компоненты (shadcn/ui)
  - **`layout/`** - компоненты макета (Header, Sidebar, Hero)
  - **`challenge/`** - компоненты для работы с челленджами
  - **`team/`** - компоненты для работы с командами
  - **`workspace/`** - компоненты рабочего пространства
  - **`mountain/`** - 3D визуализация горы для лидерборда
  - **`providers/`** - провайдеры контекста
- **`lib/`** - утилиты и библиотеки
  - **`api/`** - API клиенты для взаимодействия с сервером
  - **`stores/`** - Zustand stores для управления состоянием
  - **`utils.ts`** - вспомогательные функции
- **`types/`** - глобальные типы TypeScript
- **`public/`** - статические файлы (изображения, иконки)

### Маршрутизация

Next.js App Router использует файловую систему для маршрутизации:

- **`/`** - главная страница (landing page)
- **`/login`** - страница входа
- **`/register`** - страница регистрации
- **`/profile`** - профиль пользователя
- **`/teams`** - список команд
- **`/teams/[teamId]`** - страница команды
- **`/teams/[teamId]/challenges/[challengeId]`** - страница челленджа
- **`/challenges`** - галерея всех челленджей
- **`/settings`** - настройки пользователя

## Система аутентификации и авторизации

### AuthProvider - Провайдер аутентификации

`AuthProvider` является оберткой вокруг всего приложения, которая контролирует доступ к защищенным маршрутам. Провайдер работает следующим образом:

При монтировании компонента провайдер проверяет состояние аутентификации из `authStore`. Если пользователь аутентифицирован, выполняется запрос к серверу для проверки валидности токена через эндпоинт `/auth/me`. Если токен невалиден или истек, пользователь автоматически выходит из системы и перенаправляется на страницу входа.

Если пользователь не аутентифицирован и пытается получить доступ к защищенному маршруту (не входящему в список публичных путей), он автоматически перенаправляется на страницу входа.

Провайдер ожидает полной гидратации состояния из localStorage перед выполнением проверок, чтобы избежать мигания контента и неправильных редиректов.

### Процесс аутентификации

Когда пользователь вводит свои учетные данные на странице входа, приложение отправляет POST запрос на `/auth/login` с email и паролем. Сервер проверяет учетные данные и возвращает объект с данными пользователя, access token и refresh token.

После успешного входа, данные сохраняются в `authStore` через метод `setAuth`, который обновляет состояние и сохраняет токены в localStorage через механизм persist middleware Zustand.

Access token автоматически добавляется ко всем последующим запросам к API через заголовок `Authorization: Bearer <token>`. Если сервер возвращает статус 401 (Unauthorized), API клиент автоматически пытается обновить токен, используя refresh token. Если обновление успешно, запрос повторяется с новым токеном. Если обновление не удалось, пользователь выходит из системы.

### Защита маршрутов

Все страницы в директории `(dashboard)/` защищены через `AuthProvider`. Если пользователь не аутентифицирован, он не сможет получить доступ к этим страницам - произойдет автоматический редирект на страницу входа.

Layout компонент `(dashboard)/layout.tsx` дополнительно проверяет состояние аутентификации и показывает индикатор загрузки, пока проверка не завершена. Это предотвращает мигание контента и обеспечивает плавный пользовательский опыт.

## Управление состоянием (Zustand Stores)

Приложение использует Zustand для управления глобальным состоянием. Zustand - это легковесная библиотека, которая предоставляет простой и эффективный способ управления состоянием без излишней сложности.

### AuthStore - Хранилище аутентификации

`authStore` управляет состоянием аутентификации пользователя. Store содержит следующую информацию:

- **user** - объект с данными текущего пользователя (id, email, nickname, avatarUrl, bio, headerBgUrl)
- **accessToken** - JWT токен для доступа к API
- **refreshToken** - токен для обновления access token
- **isAuthenticated** - булево значение, указывающее, аутентифицирован ли пользователь
- **isLoading** - флаг загрузки (используется при проверке аутентификации)
- **isHydrated** - флаг, указывающий, что состояние было восстановлено из localStorage

Store предоставляет следующие методы:

- **setAuth(user, accessToken, refreshToken)** - устанавливает данные аутентификации после успешного входа или регистрации
- **setUser(user)** - обновляет только данные пользователя (например, после редактирования профиля)
- **logout()** - очищает все данные аутентификации и выходит из системы
- **setLoading(loading)** - устанавливает флаг загрузки
- **setHydrated(hydrated)** - устанавливает флаг гидратации

Store использует middleware `persist` для автоматического сохранения состояния в localStorage. Это означает, что при перезагрузке страницы состояние аутентификации восстанавливается автоматически. Ключ для хранения в localStorage - `momentum-auth`.

При восстановлении состояния из localStorage срабатывает callback `onRehydrateStorage`, который устанавливает флаги `isHydrated` и `isLoading` в соответствующие значения.

### ThemeStore - Хранилище темы и фона

`themeStore` управляет настройками темы и фонового изображения приложения. Store содержит:

- **theme** - текущая тема ('light', 'dark', 'system')
- **backgroundMode** - режим фона ('dynamic', 'static', 'photo')
- **backgroundShape** - форма 3D объекта для динамического фона ('torus', 'cube', 'pyramid', 'sphere')
- **backgroundImage** - URL изображения для фонового режима 'photo'

Store предоставляет методы для изменения каждого из этих параметров. Состояние также сохраняется в localStorage с ключом `momentum-theme`.

Настройки темы синхронизируются с настройками пользователя на сервере через API `/user-settings`. При загрузке страницы профиля настройки загружаются с сервера и применяются к themeStore, а при изменении настроек они сохраняются как локально, так и на сервере.

### UIStore - Хранилище UI состояния

`uiStore` управляет состоянием пользовательского интерфейса, таким как конфигурация камеры для 3D сцены, кастомные фоновые изображения для Hero компонента и другие UI-специфичные настройки.

Store позволяет устанавливать кастомные фоновые изображения для Hero компонента на страницах команд, что создает уникальный визуальный опыт для каждой команды.

## API клиент и взаимодействие с сервером

### Базовый API клиент (client.ts)

Базовый модуль `client.ts` предоставляет функцию `apiRequest`, которая является оберткой над стандартным `fetch` API. Эта функция обрабатывает все HTTP запросы к серверу и обеспечивает единообразную обработку ошибок и автоматическое обновление токенов.

Функция `apiRequest` принимает endpoint (путь относительно базового URL) и опции запроса (метод, тело запроса, заголовки, флаг пропуска аутентификации). Перед отправкой запроса функция автоматически добавляет access token в заголовок Authorization, если пользователь аутентифицирован и не указан флаг `skipAuth`.

Если сервер возвращает статус 401 (Unauthorized), функция автоматически пытается обновить токен через вызов `refreshAccessToken`. Эта функция отправляет POST запрос на `/auth/refresh` с refresh token. Если обновление успешно, новые токены сохраняются в authStore, и исходный запрос повторяется с новым access token. Если обновление не удалось, пользователь автоматически выходит из системы, и функция возвращает ошибку.

Все запросы используют `credentials: 'include'` для поддержки CORS с credentials. Тело запроса автоматически сериализуется в JSON, если оно предоставлено.

Функция возвращает объект с полями `data` и `error`. Если запрос успешен, `data` содержит ответ сервера, а `error` равен null. Если произошла ошибка, `data` равен null, а `error` содержит сообщение об ошибке.

### Специализированные API клиенты

Каждый модуль приложения имеет свой собственный API клиент, который использует базовую функцию `apiRequest`:

#### AuthApi - Аутентификация

`authApi` предоставляет методы для работы с аутентификацией:

- **register(data)** - регистрация нового пользователя. После успешной регистрации автоматически устанавливает данные аутентификации в authStore.
- **login(data)** - вход в систему. После успешного входа автоматически устанавливает данные аутентификации в authStore.
- **logout()** - выход из системы. Отправляет запрос на сервер для инвалидации refresh token, затем очищает локальное состояние.
- **getMe()** - получение данных текущего пользователя.
- **getSessions()** - получение списка всех активных сессий пользователя.
- **revokeSession(sessionId)** - отзыв конкретной сессии.
- **logoutAll()** - выход из всех устройств. Отправляет запрос на сервер для удаления всех сессий, затем очищает локальное состояние.

#### TeamsApi - Команды

`teamsApi` предоставляет полный набор методов для работы с командами:

- **getAll()** - получение всех публичных команд.
- **getMyTeams()** - получение команд, в которых пользователь является членом.
- **getById(teamId)** - получение информации о команде по ID.
- **create(data)** - создание новой команды.
- **update(teamId, data)** - обновление настроек команды.
- **delete(teamId)** - удаление команды (только для владельца).
- **join(teamId)** - присоединение к публичной команде.
- **leave(teamId)** - покидание команды.
- **getStatistics(teamId)** - получение статистики команды.
- **getMembers(teamId)** - получение списка членов команды.
- **updateMemberRole(teamId, userId, role)** - изменение роли члена команды.
- **removeMember(teamId, userId)** - удаление члена из команды.
- **transferOwnership(teamId, newOwnerId)** - передача владения командой.
- **getMemberStatistics(teamId, userId)** - получение статистики члена команды.

Методы для работы с кодами приглашений:

- **createInvite(teamId, data)** - создание нового кода приглашения.
- **getInvites(teamId)** - получение всех кодов приглашений команды.
- **validateInviteCode(code)** - валидация кода приглашения и получение информации о команде.
- **joinWithCode(code)** - присоединение к команде по коду приглашения.
- **deactivateInvite(teamId, inviteId)** - деактивация кода приглашения.
- **deleteInvite(teamId, inviteId)** - удаление кода приглашения.

Методы для работы с whitelist email:

- **getWhitelist(teamId)** - получение списка whitelist email.
- **addWhitelistEmail(teamId, email)** - добавление email в whitelist.
- **addBulkWhitelistEmails(teamId, emails)** - массовое добавление email в whitelist.
- **removeWhitelistEmail(teamId, whitelistId)** - удаление email из whitelist.

#### ChallengesApi - Челленджи

`challengesApi` предоставляет методы для работы с челленджами:

- **getByTeam(teamId, status?)** - получение всех челленджей команды с опциональной фильтрацией по статусу ('active', 'upcoming', 'past').
- **getById(teamId, challengeId)** - получение информации о челлендже.
- **create(teamId, data)** - создание нового челленджа.
- **update(teamId, challengeId, data)** - обновление челленджа.
- **delete(teamId, challengeId)** - удаление челленджа.
- **getStatistics(teamId, challengeId)** - получение статистики челленджа.
- **getLeaderboard(teamId, challengeId)** - получение таблицы лидеров челленджа.

#### HabitsApi и HabitLogsApi - Привычки и логи

`habitsApi` предоставляет методы для работы с привычками:

- **getByChallenge(challengeId)** - получение всех привычек челленджа.
- **getById(challengeId, habitId)** - получение информации о привычке.
- **create(challengeId, data)** - создание новой привычки.
- **update(challengeId, habitId, data)** - обновление привычки.
- **delete(challengeId, habitId)** - удаление привычки.
- **getStatistics(challengeId, habitId)** - получение статистики привычки.
- **getLeaderboard(challengeId, habitId)** - получение таблицы лидеров для конкретной привычки.

`habitLogsApi` предоставляет методы для работы с логами выполнения привычек:

- **createOrUpdate(data)** - создание или обновление лога выполнения привычки.
- **bulkUpdate(logs)** - массовое обновление логов (для эффективного редактирования таблицы).
- **delete(logId)** - удаление лога.
- **getByHabitAndUser(habitId, userId, startDate?, endDate?)** - получение логов пользователя для конкретной привычки в диапазоне дат.
- **getChallengeTable(challengeId, userId?)** - получение всех логов челленджа, организованных для отображения в таблице. Возвращает структурированные данные с логами, сгруппированными по привычкам, датам и пользователям.
- **getTodayHabits()** - получение всех привычек, которые нужно выполнить сегодня.
- **getHistory(days?)** - получение истории выполнения привычек пользователя за указанное количество дней.

#### PostsApi - Посты

`postsApi` предоставляет методы для работы с постами в командах:

- **getByTeam(teamId)** - получение всех постов команды.
- **getById(teamId, postId)** - получение информации о посте.
- **create(teamId, data)** - создание нового поста.
- **update(teamId, postId, data)** - обновление поста.
- **delete(teamId, postId)** - удаление поста.

#### ChatApi - Чат

`chatApi` предоставляет методы для работы с чатом команды:

- **getChat(teamId)** - получение информации о чате команды.
- **getMessages(teamId, limit?, cursor?)** - получение сообщений чата с пагинацией. Поддерживает курсорную пагинацию для эффективной загрузки больших списков сообщений.
- **sendMessage(teamId, data)** - отправка нового сообщения в чат.
- **deleteMessage(teamId, messageId)** - удаление сообщения.

#### UsersApi - Пользователи

`usersApi` предоставляет методы для работы с профилем пользователя:

- **getProfile()** - получение профиля текущего пользователя.
- **updateProfile(data)** - обновление профиля пользователя.
- **getPublicProfile(nickname)** - получение публичного профиля пользователя по никнейму.
- **changePassword(data)** - изменение пароля пользователя.
- **getStatistics()** - получение статистики пользователя (количество команд, выполненных привычек, челленджей, текущий streak).
- **getDashboard()** - получение конфигурации дашборда пользователя.
- **updateDashboard(layoutConfig)** - обновление конфигурации дашборда.
- **getSettings()** - получение настроек пользователя (тема, язык, расположение виджетов, фон).
- **updateSettings(settings)** - обновление настроек пользователя.

### Обработка ошибок

Все API методы возвращают объект с полями `data` и `error`. Если запрос успешен, `data` содержит ответ сервера, а `error` равен null. Если произошла ошибка, `data` равен null, а `error` содержит сообщение об ошибке.

Сообщения об ошибках могут быть строкой или массивом строк (если сервер возвращает несколько ошибок валидации). API клиент автоматически обрабатывает массивы ошибок, объединяя их в одну строку.

При сетевых ошибках (например, отсутствие интернет-соединения) API клиент возвращает общее сообщение "Network error".

### Автоматическое обновление токенов

Одной из ключевых особенностей API клиента является автоматическое обновление истекших access token. Когда сервер возвращает статус 401, клиент автоматически пытается обновить токен, используя refresh token. Если обновление успешно, исходный запрос повторяется с новым токеном. Это обеспечивает прозрачное обновление токенов без необходимости ручного вмешательства со стороны компонентов.

## Страницы приложения

### Главная страница (/) - Landing Page

Главная страница является публичной и доступна всем пользователям, включая неаутентифицированных. Страница служит точкой входа в приложение и предоставляет информацию о продукте.

Страница состоит из нескольких секций:

**Hero Section** - главная секция с заголовком и призывом к действию. Отображает название приложения, краткое описание и кнопки для регистрации или входа. Если пользователь уже аутентифицирован, отображается кнопка для перехода в дашборд.

В центре hero секции отображается визуальная превью приложения - сетка из цветных квадратов, имитирующая habit tracking grid. Это дает пользователям визуальное представление о том, как работает отслеживание привычек.

**Features Section** - секция с описанием основных функций приложения. Отображает шесть карточек с иконками, каждая из которых описывает одну из ключевых функций: Team Challenges, Track Progress, Detailed Analytics, Team Chat, Gamification, Custom Dashboard.

**CTA Section** - секция с призывом к действию, побуждающая пользователей зарегистрироваться.

**Footer** - подвал страницы с информацией о приложении и копирайтом.

Страница адаптивна и корректно отображается на всех размерах экранов. На мобильных устройствах некоторые элементы могут быть скрыты или перестроены для лучшего пользовательского опыта.

### Страница входа (/login)

Страница входа предоставляет форму для аутентификации существующих пользователей. Страница разделена на две колонки: левая содержит форму входа, правая (только на десктопе) отображает Hero компонент с 3D визуализацией.

Форма входа содержит следующие поля:

- **Email** - поле для ввода email адреса пользователя
- **Password** - поле для ввода пароля с возможностью показать/скрыть пароль

Форма также содержит ссылку "Forgot your password?" (в настоящее время не реализована) и кнопку для входа через GitHub (заглушка).

При отправке формы выполняется валидация полей (email должен быть валидным, пароль обязателен). Если валидация прошла успешно, отправляется запрос на `/auth/login` через `authApi.login()`. Если вход успешен, данные аутентификации сохраняются в authStore, отображается уведомление об успешном входе, и пользователь перенаправляется на страницу профиля.

Если вход не удался (неверные учетные данные или ошибка сети), отображается уведомление об ошибке с описанием проблемы.

Внизу формы есть ссылка для перехода на страницу регистрации для новых пользователей.

### Страница регистрации (/register)

Страница регистрации похожа на страницу входа по структуре, но содержит форму для создания нового аккаунта. Форма содержит следующие поля:

- **Email** - email адрес для регистрации
- **Password** - пароль (должен соответствовать требованиям безопасности)
- **Nickname** - уникальное имя пользователя

При отправке формы выполняется валидация всех полей. Email должен быть валидным, пароль должен быть достаточно сложным, nickname должен быть уникальным и соответствовать определенным правилам (например, минимальная длина).

Если валидация прошла успешно, отправляется запрос на `/auth/register` через `authApi.register()`. Если регистрация успешна, данные аутентификации сохраняются в authStore, отображается уведомление об успешной регистрации, и пользователь перенаправляется на страницу профиля.

Если регистрация не удалась (например, email или nickname уже заняты), отображается уведомление об ошибке с описанием проблемы.

### Страница профиля (/profile)

Страница профиля является главной страницей дашборда пользователя. Это персонализированное рабочее пространство, где пользователь может видеть свою статистику, управлять виджетами и получать быстрый доступ к своим командам.

Страница состоит из следующих элементов:

**Заголовок профиля** - отображает аватар пользователя, приветственное сообщение с никнеймом и мотивационный текст. Аватар может быть загружен пользователем или отображает первую букву никнейма.

**Рабочее пространство (Desktop Workspace)** - на десктопе страница использует компонент `DesktopWorkspace`, который предоставляет интерактивное рабочее пространство с возможностью перетаскивания виджетов. Виджеты могут быть размещены в трех локациях:

- **Placeholder slots** - фиксированные слоты в верхней части страницы (обычно 4-6 слотов)
- **Workspace** - свободная область, где виджеты можно размещать в произвольных позициях
- **Dock** - нижняя панель для дополнительных виджетов

Виджеты можно перетаскивать между локациями, изменять их размер и позицию. Все изменения автоматически сохраняются на сервере через debounce механизм (задержка 500ms после последнего изменения).

**Мобильное представление** - на мобильных устройствах виджеты отображаются в виде простого списка без возможности перетаскивания. Виджеты располагаются в фиксированном порядке.

**Доступные виджеты:**

1. **Current Streak** - отображает текущую серию последовательных дней с выполнением привычек. Виджет показывает число дней и иконку пламени.

2. **Completed Habits** - отображает общее количество выполненных привычек за все время.

3. **Active Teams** - отображает количество команд, в которых пользователь является членом.

4. **Challenges** - отображает количество челленджей, в которых пользователь участвует.

5. **My Teams** - виджет со списком команд пользователя. Отображает до 5 команд с аватарами, названиями и количеством членов. Каждая команда является кликабельной ссылкой для перехода на страницу команды. Если команд больше 5, отображается кнопка "View all teams".

При загрузке страницы выполняется параллельная загрузка статистики пользователя, списка команд и настроек пользователя. Настройки применяются к themeStore и определяют расположение виджетов и настройки фона.

Изменения в расположении виджетов сохраняются на сервере через API `/user-settings` с использованием debounce для минимизации количества запросов.

### Страница списка команд (/teams)

Страница отображает все команды, доступные пользователю. Страница разделена на две вкладки:

**Мои команды** - отображает все команды, в которых пользователь является членом. Команды отображаются в виде карточек с информацией о команде: название, слоган, описание, количество членов, количество челленджей. Каждая карточка содержит кнопку "Перейти" для перехода на страницу команды.

Если у пользователя нет команд, отображается empty state с предложением создать команду или присоединиться к существующей.

**Найти команды** - отображает все публичные команды, к которым пользователь может присоединиться. Команды, в которых пользователь уже является членом, фильтруются и не отображаются.

Вверху вкладки "Найти команды" находится поле поиска, которое позволяет фильтровать команды по названию или слогану.

Каждая карточка команды показывает:
- Название команды
- Слоган (если есть)
- Описание (если есть, обрезается до 2 строк)
- Количество членов
- Количество челленджей
- Индикатор приватности (иконка замка для приватных команд)
- Badge "Требуется код" для команд, которые требуют код приглашения

Для публичных команд без требования кода отображается кнопка "Присоединиться", которая позволяет пользователю присоединиться к команде одним кликом. Для команд с требованием кода отображается кнопка "Требуется код приглашения" (неактивная).

В верхней части страницы находятся две кнопки:
- **Ввести код** - открывает диалог для ввода 6-значного кода приглашения. При вводе кода выполняется автоматическая валидация после ввода всех 6 символов. Если код валиден, отображается информация о команде (название, слоган, количество членов и челленджей). Пользователь может затем присоединиться к команде по коду.
- **Создать команду** - перенаправляет на страницу создания новой команды.

### Страница команды (/teams/[teamId])

Страница команды является центральным местом для взаимодействия с командой. Страница отображает подробную информацию о команде, ее членах, челленджах, постах и статистике.

**Заголовок команды** - отображает название команды крупным шрифтом, слоган (если есть), описание (если есть), количество членов и челленджей. В заголовке также находятся кнопки для открытия чата, редактирования команды (для админов/владельцев) и выхода из команды (для обычных членов).

Для владельца команды доступна опция удаления команды через выпадающее меню настроек.

**Poster Rail Block** - компонент, отображающий челленджи команды в виде горизонтальной прокручиваемой ленты с постерами. Постеры челленджей отображаются в виде карточек с изображениями (или градиентными фонами, если постер не загружен). При наведении на постер он увеличивается и выдвигается вперед, создавая эффект глубины.

**Основной контент** - разделен на две колонки на больших экранах:

**Левая колонка:**
- **Chart Block** - график активности команды, показывающий количество выполненных привычек за выбранный период (неделя, месяц, год). График обновляется при изменении периода через переключатель.
- **Post List** - список постов команды с возможностью создания новых постов, редактирования и удаления существующих.

**Правая колонка:**
- **Poster Circle Block** - круговая визуализация челленджей команды. Челенджи отображаются в виде круговых карточек, расположенных по кругу. При клике на карточку происходит переход на страницу челленджа.
- **Мотивационный текст** - блок с вдохновляющим текстом о команде и привычках.
- **Radar Chart Block** - радарный график, показывающий активность по недавним челленджам. График отображает процент выполнения для каждого челленджа.
- **User Marquee Block** - бегущая строка с аватарами участников команды. Отображает всех членов команды в виде непрерывно прокручивающейся ленты.
- **Список участников** - таблица с участниками команды, отображающая аватары, никнеймы и роли (OWNER, ADMIN, MEMBER). Для владельца отображается иконка короны, для админов - иконка щита.

**Team Chat** - чат команды доступен через кнопку "Chat" в заголовке. Чат открывается в виде Sheet (боковая панель на десктопе, нижняя панель на мобильных). Чат отображает сообщения команды с возможностью отправки новых сообщений, редактирования и удаления своих сообщений (или любых сообщений для админов).

При загрузке страницы выполняется параллельная загрузка информации о команде, списка членов, челленджей и статистики. Если команда имеет кастомное фоновое изображение (headerBgUrl), оно устанавливается как фон Hero компонента через uiStore.

Страница сохраняет выбранный челлендж в настройках пользователя, чтобы при следующем посещении автоматически открывался последний просмотренный челлендж.

### Страница челленджа (/teams/[teamId]/challenges/[challengeId])

Страница челленджа является центральным местом для отслеживания прогресса в челлендже. Страница отображает подробную информацию о челлендже, таблицу отслеживания привычек, статистику и лидерборд.

**Заголовок челленджа** - отображает название челленджа, описание (если есть), статус (Активен, Скоро, Завершён), даты начала и окончания, количество привычек и участников. В заголовке также находятся кнопки для создания новой привычки (для админов), редактирования челленджа (для админов) и удаления челленджа (для админов).

**Habit Tracker Table** - основная таблица отслеживания привычек. Таблица отображает сетку, где строки представляют привычки, а столбцы - даты в периоде челленджа. Каждая ячейка представляет лог выполнения привычки конкретным пользователем на конкретную дату.

Ячейки могут иметь следующие состояния:
- **Пустая** - привычка не выполнена
- **Зеленая** - привычка полностью выполнена (COMPLETED)
- **Желтая** - привычка частично выполнена (PARTIAL)
- **Серая** - привычка пропущена с разрешением (SKIPPED_EXCUSED, только для админов)

При клике на ячейку открывается меню для изменения статуса. Обычные пользователи могут устанавливать статусы COMPLETED и PARTIAL только для своих собственных привычек. Админы могут устанавливать статус SKIPPED_EXCUSED для любых пользователей.

При клике на название привычки происходит переход на страницу детальной информации о привычке.

Таблица поддерживает фильтрацию по пользователю - можно выбрать конкретного участника для просмотра только его логов.

**Mountain Leaderboard** - 3D визуализация лидерборда в виде горы. Участники отображаются как точки на поверхности горы, где высота соответствует их проценту выполнения. Топ-5 участников отображаются с рангами. Гора автоматически вращается для лучшего обзора.

**Leaderboard Table** - таблица лидерборда с детальной информацией о каждом участнике. Таблица показывает позицию, аватар и никнейм участника, количество выполненных привычек, общее количество логов и процент выполнения. Текущий пользователь выделяется меткой "(вы)".

**Статистические карточки** - четыре карточки с ключевыми метриками:
- **Прогресс** - количество прошедших дней от общего количества дней челленджа с прогресс-баром
- **Выполнение** - процент выполнения с количеством выполненных привычек
- **Частичные** - количество частично выполненных привычек
- **Пропущенные** - количество пропущенных с разрешением привычек

При загрузке страницы выполняется параллельная загрузка информации о челлендже, команде, статистике и данных таблицы. Данные таблицы загружаются отдельно, так как они могут быть большими и требуют времени на обработку.

При обновлении логов (изменении статуса в таблице) выполняется обновление данных таблицы для отображения актуальной информации. Обновление выполняется с индикатором загрузки для лучшего пользовательского опыта.

### Страница галереи челленджей (/challenges)

Страница галереи челленджей предоставляет визуальный обзор всех челленджей пользователя из всех его команд. Страница имеет уникальный дизайн с 3D эффектами и анимациями.

**Hero Section** - верхняя секция с 3D визуализацией постеров челленджей. Постеры отображаются в виде "облака" карточек, расположенных в 3D пространстве. Карточки плавно дрейфуют и вращаются, создавая эффект глубины. В центре hero секции отображается логотип приложения.

Hero секция имеет градиентный фон и эффекты свечения для создания атмосферного визуального опыта. При прокрутке страницы hero секция плавно исчезает (fade out эффект).

**Grid Section** - основная секция с сеткой челленджей. При прокрутке страницы сетка плавно появляется (fade in эффект). Сетка отображает все челленджи пользователя в виде карточек с постерами (или градиентными фонами, если постер не загружен).

Каждая карточка челленджа показывает:
- Постер или градиентный фон
- Название челленджа
- Описание (если есть, обрезается до 2 строк)
- Статус (Активно, Скоро, Завершено) в виде badge
- Название команды

При наведении на карточку она поднимается и увеличивается тень, создавая эффект глубины.

Вверху grid секции находится поле поиска, которое позволяет фильтровать челленджи по названию, описанию или названию команды.

При клике на карточку челленджа происходит переход на страницу детальной информации о челлендже.

Если у пользователя нет челленджей, отображается empty state с предложением создать первый челлендж.

### Страница настроек (/settings)

Страница настроек позволяет пользователю управлять своими предпочтениями и настройками приложения. Страница содержит следующие секции:

**Настройки темы** - переключатель между светлой, темной и системной темами. Системная тема автоматически определяет тему на основе настроек операционной системы.

**Настройки фона** - выбор режима фона:
- **Dynamic** - динамический 3D фон с анимированными объектами (тор, куб, пирамида, сфера)
- **Static** - статический фон без анимации
- **Photo** - фоновое изображение

Для режима Photo можно загрузить собственное изображение или выбрать из предустановленных.

**Настройки языка** - выбор языка интерфейса (в настоящее время поддерживается английский и русский).

**Настройки уведомлений** - управление уведомлениями (в настоящее время не реализовано).

**Управление аккаунтом** - секция с опциями для изменения пароля, удаления аккаунта и других действий, связанных с аккаунтом.

Все изменения настроек автоматически сохраняются на сервере и применяются к приложению в реальном времени.

## Компоненты и их назначение

### Layout компоненты

#### Header - Шапка приложения

Header отображается на всех страницах приложения и содержит:
- Логотип приложения (кликабельный, ведет на главную страницу)
- Навигационное меню с ссылками на основные разделы
- Аватар пользователя с выпадающим меню (профиль, настройки, выход)

На мобильных устройствах навигационное меню может быть скрыто в гамбургер-меню.

#### Sidebar - Боковая панель

Sidebar отображается на страницах дашборда и содержит навигационные ссылки на основные разделы приложения. На мобильных устройствах sidebar может быть скрыт или отображаться как выдвижная панель.

#### Hero - Фоновый компонент

Hero компонент отображает 3D сцену с анимированными объектами в качестве фона. Компонент использует Three.js и React Three Fiber для рендеринга 3D графики. Форма объекта (тор, куб, пирамида, сфера) и режим анимации (динамический или статический) определяются настройками пользователя из themeStore.

Hero компонент может отображать кастомное фоновое изображение для страниц команд, установленное через uiStore.

#### ContentWrapper - Обертка контента

ContentWrapper обеспечивает правильные отступы и центрирование контента на всех страницах.

### Challenge компоненты

#### HabitTrackerTable - Таблица отслеживания привычек

HabitTrackerTable является центральным компонентом для отслеживания прогресса в челлендже. Компонент отображает сетку, где строки представляют привычки, а столбцы - даты.

Компонент поддерживает:
- Интерактивное редактирование статусов через клик на ячейки
- Фильтрацию по пользователю
- Визуальную индикацию статусов цветами
- Плавные анимации при изменении статусов
- Оптимизацию производительности для больших таблиц через виртуализацию

Компонент принимает данные в структурированном формате, где логи организованы по привычкам, датам и пользователям. Это позволяет эффективно отображать и обновлять данные.

#### PosterCircleBlock - Круговая визуализация челленджей

PosterCircleBlock отображает челленджи команды в виде круговых карточек, расположенных по кругу. Карточки автоматически позиционируются по кругу с равномерным распределением. При клике на карточку происходит переход на страницу челленджа.

#### PosterRailBlock - Горизонтальная лента постеров

PosterRailBlock отображает челленджи в виде горизонтальной прокручиваемой ленты. Постеры отображаются в виде стопки карточек с эффектом глубины. При наведении на постер он увеличивается и выдвигается вперед.

#### ChallengePosterGrid - Сетка постеров

ChallengePosterGrid отображает челленджи в виде адаптивной сетки карточек. Сетка автоматически подстраивается под размер экрана.

### Team компоненты

#### PostList - Список постов

PostList отображает все посты команды в хронологическом порядке (новые первыми). Каждый пост показывает автора, дату создания, заголовок, содержимое и медиафайлы (если есть).

Компонент поддерживает:
- Создание новых постов через форму
- Редактирование своих постов
- Удаление своих постов или любых постов (для админов)
- Отображение медиафайлов (изображения, видео)

#### CreatePostForm - Форма создания поста

CreatePostForm предоставляет интерфейс для создания нового поста в команде. Форма содержит поля для заголовка, содержимого и загрузки медиафайлов.

#### TeamChat - Чат команды

TeamChat отображает сообщения команды в виде Sheet (боковая панель на десктопе, нижняя панель на мобильных). Компонент поддерживает:
- Отображение сообщений с аватарами и никнеймами авторов
- Отправку новых сообщений
- Редактирование своих сообщений
- Удаление своих сообщений или любых сообщений (для админов)
- Пагинацию сообщений (загрузка старых сообщений при прокрутке)

#### ChartBlock - График активности

ChartBlock отображает график активности команды за выбранный период (неделя, месяц, год). График показывает количество выполненных привычек по дням. Компонент использует Recharts для построения графика.

#### RadarChartBlock - Радарный график

RadarChartBlock отображает радарный график активности по челленджам. График показывает процент выполнения для каждого челленджа в виде многоугольника. Компонент использует Recharts для построения графика.

#### UserMarqueeBlock - Бегущая строка участников

UserMarqueeBlock отображает участников команды в виде непрерывно прокручивающейся ленты с аватарами. Компонент создает эффект бесконечной прокрутки с плавной анимацией.

### Mountain компоненты

#### MountainLeaderboard - 3D визуализация лидерборда

MountainLeaderboard отображает лидерборд в виде 3D горы, где участники представлены как точки на поверхности. Высота точки соответствует проценту выполнения участника. Топ-5 участников отображаются с рангами.

Компонент использует Three.js и React Three Fiber для рендеринга 3D сцены. Гора генерируется алгоритмически на основе данных участников. Поверхность горы создается через контурную генерацию, где каждый участник создает "подъем" на горе пропорционально своему проценту выполнения.

Гора автоматически вращается для лучшего обзора, но вращение можно отключить. Компонент поддерживает кастомизацию цветов (фон, контур, путь, поверхность).

### Workspace компоненты

#### DesktopWorkspace - Рабочее пространство

DesktopWorkspace предоставляет интерактивное рабочее пространство для управления виджетами на странице профиля. Компонент поддерживает:

- Перетаскивание виджетов между локациями (placeholder slots, workspace, dock)
- Изменение размера виджетов
- Изменение позиции виджетов в workspace
- Автоматическое сохранение изменений на сервер

Компонент использует @dnd-kit для реализации drag-and-drop функциональности. Виджеты могут быть размещены в трех типах локаций:

- **Placeholder slots** - фиксированные слоты в верхней части страницы. Виджеты в этих слотах имеют фиксированный размер и позицию.
- **Workspace** - свободная область, где виджеты можно размещать в произвольных позициях. Виджеты в workspace могут быть перемещены и изменены в размере.
- **Dock** - нижняя панель для дополнительных виджетов. Виджеты в dock имеют фиксированную высоту, но могут быть переупорядочены.

Изменения в расположении виджетов автоматически сохраняются на сервер через debounce механизм (задержка 500ms после последнего изменения).

#### Widgets - Виджеты

Виджеты - это переиспользуемые компоненты, которые отображают различную информацию на странице профиля. Каждый виджет имеет уникальный ID, заголовок, иконку и функцию рендеринга.

Виджеты могут быть статическими (отображают только информацию) или интерактивными (позволяют взаимодействовать с данными).

### UI компоненты (shadcn/ui)

Приложение использует набор компонентов shadcn/ui, которые построены на основе Radix UI и стилизованы с помощью Tailwind CSS. Основные компоненты:

- **Button** - кнопки различных вариантов и размеров
- **Card** - карточки для группировки контента
- **Dialog** - модальные окна
- **Sheet** - боковые панели
- **Input** - поля ввода
- **Textarea** - многострочные поля ввода
- **Select** - выпадающие списки
- **Tabs** - вкладки
- **Table** - таблицы
- **Avatar** - аватары пользователей
- **Badge** - значки для статусов
- **Toast/Sonner** - уведомления
- **Skeleton** - индикаторы загрузки
- **Empty** - состояния пустых списков

Все компоненты полностью типизированы и поддерживают кастомизацию через пропсы и CSS классы.

## Типы данных и интерфейсы

### Типы аутентификации

**User** - объект пользователя содержит:
- id - уникальный идентификатор
- email - email адрес
- nickname - уникальное имя пользователя
- avatarUrl - URL аватара (может быть null)
- bio - биография пользователя (может быть null)
- headerBgUrl - URL фонового изображения профиля (может быть null)

**AuthResponse** - ответ от сервера при успешной аутентификации содержит:
- user - объект пользователя
- accessToken - JWT токен для доступа к API
- refreshToken - токен для обновления access token

**Session** - объект сессии содержит:
- id - идентификатор сессии
- userAgent - User-Agent браузера
- ipAddress - IP адрес
- createdAt - дата создания
- expiresAt - дата истечения

### Типы команд

**Team** - объект команды содержит:
- id - уникальный идентификатор
- name - название команды
- slogan - слоган команды (может быть null)
- description - описание команды (может быть null)
- headerBgUrl - URL фонового изображения (может быть null)
- isPrivate - является ли команда приватной
- requireInviteCode - требуется ли код приглашения
- requireEmailWhitelist - требуется ли whitelist email
- ownerId - ID владельца команды
- owner - объект владельца команды
- members - массив членов команды (опционально)
- _count - счетчики (количество членов, челленджей)

**TeamMember** - объект члена команды содержит:
- id - идентификатор записи
- role - роль в команде ('MEMBER', 'ADMIN', 'OWNER')
- joinedAt - дата вступления
- user - объект пользователя

**TeamInvite** - объект кода приглашения содержит:
- id - идентификатор
- teamId - ID команды
- code - код приглашения
- maxUses - максимальное количество использований (может быть null)
- usedCount - количество использований
- expiresAt - дата истечения (может быть null)
- isActive - активен ли код
- createdBy - ID создателя
- createdAt - дата создания

**TeamStatistics** - статистика команды содержит:
- membersCount - количество членов
- challengesCount - количество челленджей
- activeChallenges - количество активных челленджей
- totalCompletedHabits - общее количество выполненных привычек

### Типы челленджей

**Challenge** - объект челленджа содержит:
- id - уникальный идентификатор
- teamId - ID команды
- title - название челленджа
- description - описание (может быть null)
- posterUrl - URL постера (может быть null)
- startDate - дата начала
- endDate - дата окончания
- createdAt - дата создания
- habits - массив привычек (опционально)
- _count - счетчики (количество привычек)

**ChallengeStatistics** - статистика челленджа содержит:
- challengeId - ID челленджа
- totalDays - общее количество дней
- daysElapsed - количество прошедших дней
- habitsCount - количество привычек
- participantsCount - количество участников
- totalLogs - общее количество логов
- completedLogs - количество выполненных логов
- partialLogs - количество частично выполненных логов
- skippedLogs - количество пропущенных логов
- completionRate - процент выполнения

**LeaderboardEntry** - запись в лидерборде содержит:
- rank - позиция
- user - объект пользователя
- completedCount - количество выполненных привычек
- partialCount - количество частично выполненных привычек
- score - общий счет
- currentStreak - текущая серия (опционально)

### Типы привычек

**Habit** - объект привычки содержит:
- id - уникальный идентификатор
- challengeId - ID челленджа
- title - название привычки
- description - описание (может быть null)
- createdAt - дата создания
- _count - счетчики (количество логов)

**HabitLog** - объект лога выполнения содержит:

**TodayHabit** - привычка на сегодня содержит:
- habit - объект привычки

- mediaUrl - URL медиафайла (может быть null)

### Обратная связь с пользователем

## Заключение

Клиентское приложение Momentum представляет собой современное, производительное и удобное веб-приложение, построенное на лучших практиках разработки. Приложение обеспечивает отличный пользовательский опыт через интуитивный интерфейс, плавные анимации и четкую обратную связь.

Архитектура приложения обеспечивает масштабируемость, поддерживаемость и расширяемость. Использование TypeScript обеспечивает типобезопасность, а модульная структура позволяет легко добавлять новые функции и компоненты.
